# Production Readiness Audit Findings

Audit target: Docker deployment with OIDC, nginx/TLS termination, SQLite metadata, and local or S3/R2 object storage.

This pass reviewed the Rust server paths for auth/OIDC, HTTP routing, proxy handling, SQLite schema and migrations, documents/folders/admin/shares/preferences/settings/views, uploads, exports, transfer recovery, reconciliation, storage backends, Docker/Compose, README deployment guidance, and the browser transfer client. The report focuses on failure modes that could corrupt data, lose data, retain data unexpectedly, wedge production, or weaken an OIDC/TLS deployment.

## Critical

No critical issue was proven in the current worktree. I did not find a single-request path that obviously corrupts SQLite schema/data by itself, nor a long-running upload/download path that holds a SQLite transaction while streaming request or response bodies. The high-severity section does include concurrent normal-request races that can cause data loss or persistent inconsistency.

## High

### Availability - Full download and large range responses are buffered into memory

Download responses are range-aware, but they are not streaming from storage to the HTTP body. `version_download_response` parses the requested range, awaits `storage.read_location_range(...)`, verifies the returned `Vec` length, and then builds `Body::from(body)` (`vault/server/src/http.rs:2274-2315`). A plain non-range download uses `full_download_range(size)` and therefore requests the entire object (`vault/server/src/http.rs:2407-2413`). Local storage allocates a `Vec` sized to the requested range and reads it to completion (`vault/server/src/storage.rs:383-414`), local multipart range reads do the same across part files (`vault/server/src/storage.rs:581-620`), and S3/R2 `read_range` collects the full remote body before returning (`vault/server/src/storage.rs:936-960`).

The browser client reduces normal large-download pressure by probing `Range: bytes=0-0` and using 64 MiB segmented range requests for large files when the browser supports file-system streaming (`vault/client/src/lib/transferClient.js:729-849`, `vault/client/src/lib/transferClient.js:852-920`). That does not protect the server from a direct full-object GET, an authenticated client requesting one huge range, an export artifact download outside the first-party client, or a client/browser path that falls back to a full response. With the default 5 GiB upload limit, one request can attempt a multi-gigabyte allocation and OOM/restart the container.

Impact: authenticated users can take down the process with a large full download or large range request. Docker restart/recovery should protect committed SQLite metadata, but active uploads/exports can be interrupted, users see failed transfers, and repeated requests can keep production unhealthy.

Recommendation: make storage reads return streaming bodies or chunk streams, and cap individual range response size when streaming is not available. Keep range support, but stream each range in bounded chunks with backpressure instead of materializing the whole response in memory. Add a regression test that a full-size download does not allocate/read the whole file before sending the first body chunk.

### Data Integrity - Mutable current-version range downloads can mix bytes from different versions

The server exposes mutable current-version download routes: `/documents/{doc_id}/download` resolves `current_version_download` on each request, and `/documents/{doc_id}/checkout` does the same before recording the checkout/lock (`vault/server/src/http.rs:1471-1494`; `vault/server/src/documents.rs:438-466`). The response includes an ETag and the range parser honors `If-Range` by falling back to a full response when the validator differs (`vault/server/src/http.rs:2274-2287`, `vault/server/src/http.rs:2333-2364`).

The first-party segmented downloader probes with `Range: bytes=0-0`, derives total size from `Content-Range`, then starts concurrent range requests to the same URL (`vault/client/src/lib/transferClient.js:852-903`). Those segment requests send only `Range`; they do not pin the probe ETag with `If-Range`, do not compare response ETags, and do not switch the mutable current-version URL to the immutable version URL (`vault/client/src/lib/transferClient.js:729-824`). Normal document-list downloads often avoid this because view payloads include `/documents/{id}/versions/{version_id}/download` (`vault/server/src/views.rs:1198-1201`), but the fallback current-version URL remains in the client and the checkout action always downloads `/documents/{id}/checkout` (`vault/client/src/lib/itemActions.js:166-171`, `vault/client/src/lib/fileLockActions.js:139-143`).

Impact: if a new version is checked in while a large current-version or checkout download is segmented, different byte ranges can be served from different blobs under the same local file. The browser writes each successful `206` to its requested offset and closes the file without validating a whole-file hash, so the user can save a silently mixed/corrupt file. The route-level ETag support is not enough unless range clients use it or the server pins the download to an immutable version for the whole transfer.

Recommendation: make first-party downloads use immutable version URLs for all document bytes, including checkout, or make the current-version route return/redirect to a version-specific URL before range transfer starts. For segmented clients, capture the probe ETag and send `If-Range` on every segment; abort and restart if any segment returns `200` or an unexpected ETag. Add a test that changes `current_version_id` between range requests and proves the client/server cannot assemble mixed-version output.

### Availability - Export ZIP compression can buffer multi-gigabyte entries in memory

Large stored ZIP entries stream in 64 MiB ranges, but that path is only used when the compression plan is `Stored` and the entry is at least 64 MiB (`vault/server/src/exports.rs:41-43`, `vault/server/src/exports.rs:1175-1189`, `vault/server/src/exports.rs:1285-1364`). The buffered path calls `storage.read_location_bytes(...)`, verifies the full byte vector, optionally deflates it into another `Vec`, and then writes the resulting buffer (`vault/server/src/exports.rs:1218-1283`). The default compression threshold is 3 GiB of total export input, and once enabled, known-compressible entries such as `text/*` are always deflated rather than streamed (`vault/server/src/exports.rs:39-49`, `vault/server/src/exports.rs:1738-1763`, `vault/server/src/exports.rs:1807-1810`).

Impact: an authenticated user exporting a large text/log/CSV-style file, or a folder whose total export size crosses the compression threshold, can make the worker allocate the entire source object and then a second compressed buffer. With the default 5 GiB upload limit this can OOM/restart the container or stall export workers. SQLite metadata should survive the restart, but active uploads/exports are interrupted and repeated export attempts can keep production unstable.

Recommendation: stream compression and CRC calculation instead of requiring a full source buffer, or cap buffered/compressed entries to a small fixed size and force larger entries through the streaming stored path. Add tests for a large known-compressible entry proving memory stays bounded and cancellation remains responsive.

### Data Durability - Local object writes and SQLite commits are not fully crash-durable

The default production path uses SQLite WAL with `synchronous=NORMAL` (`vault/server/src/db/mod.rs:22-32`) and local object files under `/data/objects`. The local storage backend writes or links files, flushes process buffers, and renames paths, but does not call `sync_data`, `sync_all`, or fsync parent directories before database metadata is committed. Examples: `put_bytes` writes a temp file and renames it (`vault/server/src/storage.rs:250-270`), `put_file` moves/copies and renames (`vault/server/src/storage.rs:273-305`), and multipart staging flushes before rename but does not fsync (`vault/server/src/storage.rs:319-340`).

Impact: after host power loss, unsafe volume loss, VM crash, or forced container/host termination, SQLite can retain metadata for an object whose final file/manifest was not durable, or object data can survive without the DB row. Reconciliation can detect missing/corrupt referenced local objects, but it is not an automatic repair and cannot recreate lost bytes.

Recommendation: add a configurable durable local mode that fsyncs temp files, final files/manifests, and parent directories, and consider `synchronous=FULL` for deployments that prioritize correctness. Production readiness should also fail or warn when referenced local objects are missing/corrupt.

### Data Lifecycle - Permanent delete and delete TTL leave orphaned blobs/objects

`delete_document_forever` only deletes the `documents` row (`vault/server/src/documents.rs:324-349`). Delete-TTL expiry collects document IDs and also deletes only `documents` rows (`vault/server/src/documents.rs:401-428`). The schema cascades `document_versions` when a document is deleted, but blobs are intentionally separate: `document_versions.blob_id` references `blobs(id)` without cascade, and `blob_locations` only cascade when the blob row itself is deleted (`vault/server/src/db/schema.rs:95-114`, `vault/server/src/db/schema.rs:159-163`).

There is local reconciliation code that can delete orphan local keys/locations when `apply=true` (`vault/server/src/reconciliation.rs:52-63`, `vault/server/src/reconciliation.rs:240-291`), but the production HTTP route is dev-only and read-only with `apply=false` (`vault/server/src/http.rs:873-883`). Remote orphan blobs are preserved if they still have remote locations (`vault/server/src/reconciliation.rs:308-327`).

Transfer cleanup also intentionally retains content-addressed objects after metadata cleanup. `sweep_expired_transfers_with_limit` now commits upload/export row cleanup and removes only upload scratch directories and export temp files after the transaction (`vault/server/src/transfers.rs:99-123`). Interrupted export recovery likewise deletes artifact/blob metadata and temp files without deleting the underlying object key (`vault/server/src/transfers.rs:235-247`, `vault/server/src/transfers.rs:455-480`). That avoids deleting bytes another writer may be about to reference, but it leaves physical object reclamation dependent on a separate GC/reconciliation path that is not production-exposed.

Impact: "delete forever" and delete TTL are not physical deletion. Disk/object-store usage can grow without bound, and sensitive content can remain under `/data/objects` or S3/R2 after the UI/API says the document was permanently deleted.

Recommendation: implement reference-counted blob garbage collection for document delete and TTL delete, or add a production-safe admin maintenance job/CLI that removes unreferenced blobs and object locations after verifying no document/export references remain. Document the retention semantics until fixed.

### Data Integrity - Folder tree mutations can use stale pre-transaction topology

Folder move/rename validates the source path, target path, self/descendant check, subtree IDs, locks, and destination write access before opening the write transaction (`vault/server/src/folders.rs:601-644`). Inside the transaction it re-fetches or creates the target parent and updates `folders.parent_id`/`name`, but it does not re-read the live folder graph or re-run the descendant/cycle check against the current database state (`vault/server/src/folders.rs:646-660`). The schema has only a self-referential FK for `folders.parent_id`, not an acyclicity trigger or constraint (`vault/server/src/db/schema.rs:7-23`). The path/subtree helpers intentionally break cycles rather than rejecting them (`vault/server/src/folders.rs:1045-1057`, `vault/server/src/folders.rs:1060-1091`, `vault/server/src/folders.rs:1327-1347`).

Folder archive has a more destructive variant of the same stale-topology pattern. `archive_folder` computes the folder subtree, the documents in that subtree, and the archive mutations before opening the transaction (`vault/server/src/documents.rs:745-792`). It then archives only that precomputed document list and deletes the precomputed folder IDs (`vault/server/src/documents.rs:793-815`, `vault/server/src/documents.rs:1390-1432`, `vault/server/src/documents.rs:1466-1479`). Because `folders.parent_id` cascades and `documents.folder_id` also cascades (`vault/server/src/db/schema.rs:7-10`, `vault/server/src/db/schema.rs:119-122`), a concurrent request that moves a folder into the archived subtree after the precompute step but before the delete can have that moved-in folder and its documents physically deleted without being archived. The same delete can also cascade a document that was newly uploaded into, or moved directly into, one of the precomputed folder IDs after `documents_in_folders` ran; create upload inserts the document/version in its own later transaction, and document move updates `documents.folder_id` by id without coordinating with folder archive (`vault/server/src/uploads.rs:995-1037`, `vault/server/src/documents.rs:856-935`).

Folder retention updates also compute topology outside the transaction. `update_folder_retention` fetches all folders, mutates the target policy in memory, derives a subtree id list, checks access, builds a folder map, and only then opens the transaction to update the target folder and reapply TTL to the precomputed subtree ids (`vault/server/src/folders.rs:513-571`). The reapply helper selects current documents by those stale folder ids but uses the stale folder map and caller-supplied folder policy (`vault/server/src/folders.rs:1241-1286`). A concurrent folder move before the retention transaction can therefore make moved-out documents receive a delete/archive TTL from a folder they no longer live under, or make moved-in documents miss the policy.

Impact: concurrent normal user actions can leave persistent folder cycles/detached subtrees, making paths, permissions, TTL inheritance, and sidebar/content views inconsistent. The archive race is worse: document rows and versions in a moved-in folder, newly-created document, or directly moved document can be cascade-deleted even though the archive operation never snapshotted or moved them to Archive. Blob rows/object bytes may survive as orphans, but the document metadata is gone from the app. The retention race can silently attach a delete/archive policy to documents whose folder tree no longer inherits it, which can later cause unexpected archival or deletion.

Recommendation: use a single `BEGIN IMMEDIATE` transaction for folder move/rename/archive/retention topology checks and mutations. Re-read the source, destination, subtree, lock set, and candidate documents inside that transaction immediately before updating, deleting, or reapplying TTL. For archive, avoid deleting folder rows by relying on a stale ID set; either mark the exact current subtree in the same transaction, or make the delete conditional on the live subtree still matching the archived document set. For retention, compute the subtree and effective policy map inside the same transaction that writes TTL fields. Add race tests for reciprocal folder moves, archive-vs-folder/document move into subtree, archive-vs-upload into subtree, and retention-vs-move-in/out-of-subtree.

### Data Integrity - Concurrent document archive and move/rename can leave Vault rows marked as archived

Single-document archive computes the document, archive status, source path, source folder path, and `archived_access` snapshot before opening the write transaction (`vault/server/src/documents.rs:637-654`). The transaction then updates the row by `id` through `archive_document_in_tx`, setting `folder_id`, `name`, `archived_from_folder`, `archived_original_name`, and `archived_access` from that stale pre-transaction snapshot (`vault/server/src/documents.rs:1390-1432`). Document move/rename has the inverse stale-snapshot shape: it reads the document, source root, destination path, archived status, and destination access before its transaction, but the transaction only updates `folder_id`, `name`, and modified fields by `id`; it does not re-read the document or clear/reject existing archive metadata (`vault/server/src/documents.rs:856-945`).

That creates a bad interleaving. If a move/rename request reads an active Vault document, then another request archives it and commits, the stale move/rename transaction can still update the same `documents.id` back into a Vault folder while leaving `archived_from_folder`, `archived_original_name`, and `archived_access` populated from the archive request. The code decides whether a document is in Archive from the current folder root (`document_is_archive`, `vault/server/src/documents.rs:993-1001`), while the active-name unique index excludes any row where `archived_from_folder IS NOT NULL` (`vault/server/src/db/schema.rs:119-140`). A row in a Vault folder with archive metadata is therefore treated by access checks as a normal Vault document but no longer participates in the active `(folder_id, name)` uniqueness constraint.

Impact: concurrent normal user actions can persist a document row that is neither cleanly active nor cleanly archived. That can allow duplicate visible document names in the same Vault folder, make restore/archive controls disagree with stored metadata, and preserve stale archive access/source metadata on an active document. The reverse interleaving, where archive commits after a concurrent move/rename, can also archive from the wrong source path/name and with the wrong source ACL snapshot.

Recommendation: move document archive/move/rename state checks into a single `BEGIN IMMEDIATE` transaction and re-fetch the current document row, folder root, archive metadata, lock, destination folder, and duplicate-name state immediately before mutation. Guard updates with the expected state (`WHERE id = ? AND folder_id = ? AND name = ? AND archived_from_folder IS NULL`, or equivalent reloaded checks) and return a conflict/stale-state response when the row changed. Add concurrency tests for archive-vs-rename, archive-vs-move, and move/rename-after-archive proving Vault rows cannot retain archive metadata and duplicate active names remain impossible.

### Data Loss - Permanent delete can remove a document after a concurrent restore

Permanent delete checks that the target document is currently in Archive before it opens the write transaction. `delete_document_forever` fetches the document, calls `document_is_archive`, computes the path, and only then starts a transaction that deletes `documents WHERE id = ?` without re-reading the row or verifying that it still belongs to the Archive root (`vault/server/src/documents.rs:324-349`). The HTTP route defaults delete-forever to admin-only, but the setting can be relaxed; either way the stale check is still reachable by an admin (`vault/server/src/http.rs:1609-1649`, `vault/server/src/site_settings.rs:28-36`, `vault/server/src/site_settings.rs:73-84`).

Restore has the matching normal mutation that can race with that stale delete. `restore_document` preloads the archived row and metadata, then in its transaction moves the same document id back under the original Vault folder, clears `archived_from_folder`, `archived_original_name`, and `archived_access`, reapplies TTL, and commits (`vault/server/src/documents.rs:677-742`). If restore commits after delete-forever's archive check but before its delete transaction, the delete transaction still removes the now-active Vault document row. Because `document_versions` cascade from `documents`, the application loses the document metadata and all version rows even though the item had been restored.

Impact: concurrent admin workflows, or non-admin delete-forever when the site setting is relaxed, can turn an Archive cleanup action into deletion of a restored active document. Blob/object bytes may remain as orphans, but the document and version history disappear from the app.

Recommendation: make permanent delete re-fetch and re-validate the live document row inside a `BEGIN IMMEDIATE` transaction immediately before deletion. Delete only if the current folder root is still Archive and the archived metadata still matches the expected state, or guard the delete with an archive-root predicate. Add a restore-vs-delete-forever race test that proves a restored document cannot be deleted by a stale Archive cleanup request.

### Data Loss - Retention sweep can delete or archive a document after a concurrent TTL-clearing move

The retention sweeper builds its work plan before opening the write transaction. It loads all folders, path cache, expired document rows, active locks, and per-document archive/delete decisions first (`vault/server/src/documents.rs:356-407`). Only after that does it begin a transaction and execute the precomputed actions: clear expiry by id, archive by id with the stale source folder/path/name snapshot, and delete by id for delete-TTL rows (`vault/server/src/documents.rs:409-433`, `vault/server/src/documents.rs:1209-1249`). The delete action is just `DELETE FROM documents WHERE id = ?`, with no predicate that the row is still expired, still has `expiry_action = 'delete'`, still lives under the folder that supplied delete TTL, or still has no active lock (`vault/server/src/documents.rs:424-428`).

Normal document and folder moves intentionally reapply TTL inside their own transactions. Moving a document or folder out of a delete-TTL subtree clears `documents.expires_at` and `expiry_action`, and the serial route tests assert that a later sweep does not delete it (`vault/server/src/folders.rs:692-722`, `vault/server/tests/routes_documents.rs:3267-3340`, `vault/server/tests/routes_folders.rs:2469-2552`). That invariant does not hold if the sweep selected the expired row first and the rescue move commits before the sweep's transaction. The stale sweep still deletes the document row by id. The archive path has the same stale-state shape: it can archive a row after a concurrent move/rename/TTL clear and record the old source path/access snapshot.

Impact: a user/admin can move a document or folder out of a delete-TTL area and get a successful response showing the item is safe, while the background sweeper can still delete that same document moments later. Because `document_versions` cascade from `documents`, the app loses the document metadata and all version rows; object bytes may remain only as orphaned blobs. For archive-TTL, the row can be moved to Archive with stale source metadata after the live folder/name/TTL state changed, leaving a confusing recoverability and access-history state.

Recommendation: make retention selection and mutation one guarded `BEGIN IMMEDIATE` workflow, or re-fetch each candidate inside the transaction immediately before acting. Delete only with predicates such as `id = ? AND expires_at IS NOT NULL AND datetime(expires_at) <= datetime('now') AND expiry_action = 'delete'` plus a live lock check; archive only if the row is still expired, still archive-TTL, and its current folder/name/source snapshot matches the plan. Check `rows_affected` and skip stale candidates. Add race tests for sweep-vs-document-move-out-of-delete-TTL, sweep-vs-folder-move-out-of-delete-TTL, sweep-vs-rename-refreshing-expiry, and sweep-vs-lock-acquisition.

### Authorization - Every synced external group is automatically granted root read/write

OIDC and header identities sync the incoming group set into `vault_groups` and memberships on login/request (`vault/server/src/auth.rs:339-380`, `vault/server/src/auth.rs:524-599`). During that sync, every non-empty group is passed to `ensure_group_root_permissions`, which inserts `folder_permissions` rows for every root folder with `can_view = 1`, `can_read = 1`, and `can_write = 1` if the row does not already exist (`vault/server/src/auth.rs:618-630`, `vault/server/src/auth.rs:679-705`). Tests assert the behavior for header auth: a user with `Remote-Groups: vault-users, vault-admin` gets the non-admin `vault-users` group synced, and `folder_permission_count_for_group(&pool, "vault-users")` is `2`, matching the Vault and Archive roots (`vault/server/tests/auth.rs:72-98`). The OIDC callback test similarly syncs provider groups such as `artists` from the token/userinfo payload (`vault/server/tests/routes_oidc.rs:861-924`).

Those rows are effective permissions, not harmless bootstrap metadata. Non-admin folder access walks ancestors and returns the matching group permission from the first folder with permission rows (`vault/server/src/folders.rs:758-795`), so a root-level write grant is inherited by default unless a descendant permission set overrides it. The admin permission editor replaces a folder's rows by deleting all current rows and inserting the submitted set (`vault/server/src/folders.rs:445-510`), but the next OIDC/header sync for a member of a removed root group will call `ensure_group_root_permissions` again and recreate the missing root write rows (`vault/server/src/auth.rs:618-630`, `vault/server/src/auth.rs:679-705`). The admin delete path also refuses to delete any group referenced by folder permissions (`vault/server/src/admin.rs:293-318`), and tests assert "Group is used by folder permissions" for such groups (`vault/server/tests/routes_admin.rs:768-787`), so externally emitted groups can leave sticky authorization state behind. The Docker/OIDC setup surfaces `VAULT_ADMIN_GROUPS` and `VAULT_OIDC_GROUPS_CLAIM`, but README production guidance does not warn that every emitted group gets full root access (`.env.example:24-48`, `README.md:21-23`).

There is also a case-normalization split between admin-created groups and synced groups. Admin group creation checks for case-insensitive duplicates with `lower(name) = ?`, but preserves the submitted display case (`vault/server/src/admin.rs:238-286`, `vault/server/src/admin.rs:679-689`). OIDC/header group parsing lowercases every incoming group value, while `ensure_group` looks up `vault_groups` with exact `WHERE name = ?` before inserting (`vault/server/src/auth.rs:618-630`, `vault/server/src/auth.rs:811-829`, `vault/server/src/oidc.rs:379-390`). A synced `artists` group can therefore be inserted next to an existing admin-created `Artists` row. Access checks then lowercase stored group names before comparing them to the user's groups, so those two DB rows behave like the same external principal even though the admin UI and permissions table treat them as separate groups (`vault/server/src/folders.rs:758-795`).

The sync surface is also unbounded. `split_groups` and `groups_from_claim` collect every non-empty comma-delimited/header or OIDC array value after lowercasing, with no count or length cap (`vault/server/src/auth.rs:811-829`, `vault/server/src/oidc.rs:379-390`). `sync_vault_groups` then loops over the full set inside the identity `BEGIN IMMEDIATE` transaction, creating each group and root permission rows before syncing memberships (`vault/server/src/auth.rs:524-599`, `vault/server/src/auth.rs:618-657`).

Impact: in a serious OIDC deployment, a broad IdP group such as `employees`, `domain-users`, `artists`, or any group forwarded by nginx/header auth becomes a full read/write principal over both roots on first login. That can expose or mutate all documents and archive contents for users who were only meant to be authenticated, not authorized for all Vault data. Attempts to revoke the root grant by deleting the permission row can be undone by the next login/request carrying that group, and the auto-created group cannot be deleted while any recreated folder permission still references it. Case-variant duplicate rows can make the same external group appear twice in administration while either row's permissions may apply to the lowercased IdP group. Large emitted group sets can also turn a login/request identity sync into a long SQLite writer transaction that persists many groups and root permission rows.

Recommendation: do not auto-grant root write permissions for arbitrary synced external groups. Default synced groups to no access, or only grant initial access to an explicitly configured allowlist/default access group. Keep admin determination (`VAULT_ADMIN_GROUPS`) separate from data access grants, require admins to assign folder permissions deliberately, store a single canonical group identity separate from display name, and cap or allowlist synced group claim values before they create persistent database state. Add migration/tests proving existing deployments do not silently retain unintended root grants or case-variant duplicate groups after the policy change.

## Medium

### Availability - Checkout reads document bytes before acquiring the edit lock

The checkout route builds the download response before taking or validating the active document lock. `checkout_document_version` calls `checkout_version_download`, then `version_download_response`, and only afterward calls `record_checkout_event_and_lock` (`vault/server/src/http.rs:1484-1494`). The pre-read helper checks write access and loads the current version, but it does not check the active lock (`vault/server/src/documents.rs:469-475`). The lock conflict check happens later inside `record_checkout_event_and_lock`, after the storage read already completed (`vault/server/src/documents.rs:590-634`, `vault/server/src/documents.rs:1336-1344`).

Impact: a user with write access can request checkout for a document that is already locked by another user, or race another checkout, and make the server read/buffer the object before returning the lock error. For large documents this compounds the buffered-download risk above: lock conflicts can consume the same memory and storage I/O as successful downloads while producing no usable response. Under contention it also lets multiple racing checkout requests perform expensive object reads even though only one lock can win.

Recommendation: acquire or validate the checkout lock before reading object bytes, then build the response. If a later storage read fails, release or mark the lock appropriately so users are not left with a stale checkout. Add a regression test that a checkout rejected by an existing lock does not call the storage read path.

### Auth / Workflow - Checkout is a side-effecting GET in cookie-auth deployments

The checkout endpoint is registered as `GET /documents/{doc_id}/checkout` (`vault/server/src/http.rs:246-252`). A successful request records a checkout event and creates or validates an active document lock (`vault/server/src/http.rs:1484-1494`, `vault/server/src/documents.rs:590-634`). OIDC sessions are authenticated with cookies set `HttpOnly; SameSite=Lax` (`vault/server/src/http.rs:512-529`, `vault/server/src/http.rs:2751-2761`), and I did not find a CSRF token or origin check on the checkout route.

Impact: in OIDC mode, a cross-site top-level navigation to a known checkout URL can include the user's session cookie under normal `SameSite=Lax` browser behavior. That can create checkout locks and audit entries without an intentional in-app action. It does not expose response bytes to the attacker because the browser navigates/downloads in the victim context, but it can wedge collaborative editing, block retention/deletion flows that skip locked documents, and create confusing audit history.

Recommendation: make checkout a non-idempotent API action using `POST` with a CSRF token or same-origin/origin validation, then return a separate download URL/body after the lock is acquired. Keep legacy GET downloads read-only, or make any GET checkout path require an explicit one-time token generated by the app.

### Auth / Availability - Document download GETs perform storage and audit side effects in cookie-auth deployments

The current and historical document download endpoints are registered as GET routes (`vault/server/src/http.rs:243-257`). A successful current-version download authenticates the user, resolves the version, reads the requested object bytes into the response, and only then records a `download` document event plus a `document.download` state event (`vault/server/src/http.rs:1471-1481`, `vault/server/src/documents.rs:560-586`). The version-specific GET route has the same shape (`vault/server/src/http.rs:1512-1521`). OIDC sessions use `HttpOnly; SameSite=Lax` cookies (`vault/server/src/http.rs:512-529`, `vault/server/src/http.rs:2751-2761`), and I did not find an origin/CSRF check on these download GET routes.

Impact: in OIDC mode, a cross-site top-level navigation to a known document download URL can include the victim's session cookie and force Vault to perform the storage read and audit/state writes in that user's context. The attacker should not receive the file bytes because the browser is navigating/downloading as the victim, but this can pollute audit history, generate unnecessary SSE invalidations, and amplify the full-download buffering and segmented-range write-pressure risks described elsewhere. It is especially relevant because document ids are numeric and the current-version route does not require an unguessable download token.

Recommendation: keep cookie-auth GET downloads side-effect free, or require a same-origin POST/CSRF-protected action to prepare or audit a download before returning a one-time URL/token for the actual GET. At minimum, do not record audit/state rows for cross-origin navigations or unaudited range probes, and add tests that cross-site-style GETs cannot create download history or force large object reads without an explicit in-app action.

### Client Download Integrity - Segmented downloads do not validate returned ranges or byte counts

The browser downloader probes with `Range: bytes=0-0`, derives a total size from `Content-Range`, and then launches concurrent 64 MiB range fetches for large downloads (`vault/client/src/lib/transferClient.js:852-903`). Each segment only requires `response.status === 206`; it never parses that segment's `Content-Range`, compares the response ETag with the probe, verifies that the body starts at the requested range, or checks that the number of bytes read equals `segment.end - segment.start + 1` before writing at the requested file position and closing the file (`vault/client/src/lib/transferClient.js:729-849`). The UI layer treats the returned size as completion metadata and adds no independent hash or range validation (`vault/client/src/lib/useTransfers.js:199-239`).

The server's own range response path is stricter: it parses the requested range, checks the storage-read length against the requested range length, and emits `Content-Range`/ETag headers (`vault/server/src/http.rs:2274-2343`). The gap is therefore primarily at the client/proxy boundary and in future backend behavior: a misbehaving reverse proxy, interrupted-but-clean `206`, or alternate download endpoint returning a short or wrong range can be accepted as success.

Impact: a large browser download can be saved with gaps, shifted bytes, or mixed validator state while the transfer dock reports completion. Because the client does not validate a whole-file digest after segmented writes, users can trust a locally corrupted file. This is narrower than the server-side current-version race above, but it removes the last client-side protection against range/proxy faults in the nginx/TLS deployment path.

Recommendation: capture the probe ETag and send `If-Range` on every segment, reject any unexpected ETag or `Content-Range`, and require each segment body length to match the requested range before closing the writer. For document bytes, prefer immutable version URLs plus optional final SHA-256 verification from trusted metadata. Add frontend tests using mocked range responses for short, shifted, and wrong-validator `206` bodies.

### Auth / CSRF - Folder creation accepts form posts without origin checks

Most mutating API routes use JSON extractors, so a plain HTML form cannot directly submit those payloads. Folder creation is different: `/folders` is a POST route that accepts `application/x-www-form-urlencoded` through `Form<CreateFolderForm>` and then creates the folder, folder event, and state event (`vault/server/src/http.rs:142`, `vault/server/src/http.rs:757-765`, `vault/server/src/folders.rs:338-403`). The first-party client uses that exact form encoding via `URLSearchParams`, and route tests exercise successful form-encoded creation (`vault/client/src/lib/folderRequests.js:1-5`, `vault/server/tests/routes_folders.rs:320-346`). I did not find a CSRF token, `Origin`, or `Sec-Fetch-Site` check on the route.

Impact: in OIDC cookie-auth mode, `SameSite=Lax` blocks many cross-site unsafe requests, but it does not protect against same-site cross-origin/subdomain form posts. If an untrusted sibling origin exists under the same site as the Vault host, it can cause an authenticated browser to create Vault folders wherever that user has write access. That does not expose file bytes or corrupt existing document contents, but it creates persistent metadata/audit noise and can trigger sidebar/content refresh state for an action the user did not intend.

Recommendation: move folder creation to a JSON API protected by the same origin/CSRF policy as other mutating routes, or add explicit origin/fetch-metadata checks and a CSRF token for cookie-auth POSTs. Keep any legacy form route behind the same protection. Add tests that a form-style request without an accepted origin/CSRF signal is rejected in OIDC cookie mode.

### Auth Deployment - Docker Compose disables the Docker runtime session-secret requirement when the variable is empty

The image sets `VAULT_DOCKER_RUNTIME=1`, and the auth parser requires an explicit session secret when `VAULT_REQUIRE_SESSION_SECRET` is absent and Docker runtime is true (`vault/server/src/auth.rs:894-904`). However, `docker-compose.yml` always passes `VAULT_REQUIRE_SESSION_SECRET: ${VAULT_REQUIRE_SESSION_SECRET:-}` (`docker-compose.yml:27-30`). That means the environment variable is present but empty, so `env::var` succeeds, `env_value_is_truthy("")` is false, and the Docker runtime fallback is bypassed.

If `VAULT_SESSION_SECRET` is empty, `session_secret_from_env` falls back to `VAULT_OIDC_CLIENT_SECRET` before the development secret (`vault/server/src/auth.rs:876-891`). Runtime validation does reject the built-in development secret outside dev mode, so a Compose deployment with no session secret and no OIDC client secret fails closed (`vault/server/src/auth.rs:283-285`; `vault/server/tests/auth.rs:454-470`). The remaining gap is OIDC production: because the empty compose variable makes the requirement optional, startup can pass with only `VAULT_OIDC_CLIENT_SECRET` set, signing session and OIDC state cookies with the OIDC client secret despite README guidance to set a strong `VAULT_SESSION_SECRET` (`README.md:7-23`).

Impact: secret reuse and silent misconfiguration. Rotating the OIDC client secret invalidates sessions, cookie-signing material is coupled to an IdP credential, and the intended Docker fail-closed behavior is not actually enforced by the provided compose file.

Recommendation: treat an empty `VAULT_REQUIRE_SESSION_SECRET` the same as absent, set the compose default to `1`, or remove the variable from compose so `VAULT_DOCKER_RUNTIME=1` applies. Add a test for the compose-empty case.

### Auth Deployment - Explicit session secrets are not strength-checked

The runtime rejects the built-in development session secret outside dev mode and can require that `VAULT_SESSION_SECRET` be explicitly set (`vault/server/src/auth.rs:250-289`, `vault/server/src/auth.rs:876-910`). It does not validate length, entropy, or obvious weak values for an explicit secret: any non-empty trimmed string becomes the HMAC key for session cookies, OIDC state cookies, and upload part tokens (`vault/server/src/auth.rs:453-491`, `vault/server/src/uploads.rs:2160-2265`). README asks operators to set a strong secret and `.env.example` says to generate one with `openssl rand -hex 32`, but startup does not enforce that guidance (`README.md:7-23`, `.env.example:19-22`).

Impact: a typo or weak explicit value such as a short word can pass production startup and protect all signed browser sessions with guessable key material. If the secret is guessed or leaked, an attacker can forge valid session/state/token payloads within their normal expiry windows. This is configuration hardening, but it is central to OIDC/cookie-auth production safety.

Recommendation: require an explicit production session secret to meet a minimum strength policy, such as at least 32 random bytes after decoding a documented hex/base64 form, or at least 43 characters of URL-safe random text. Reject known development/default/common values and add startup tests for too-short explicit secrets.

### Auth Deployment - Compose default public URL shadows forwarded OIDC callback and share-link URLs

`docker-compose.yml` always injects `VAULT_PUBLIC_URL`, defaulting it to `http://localhost:8000` (`docker-compose.yml:9-10`). `.env.example` carries the same localhost value (`.env.example:7-8`). The OIDC redirect builder prefers any non-empty `VAULT_PUBLIC_URL` over `X-Forwarded-Host`/`X-Forwarded-Proto`, returning `{public_url}/auth/callback` before looking at forwarded headers (`vault/server/src/http.rs:2681-2696`). Share-link creation also passes `state.auth.public_url` into the share service, and `share_url` prefixes it directly when non-empty (`vault/server/src/http.rs:1068-1076`, `vault/server/src/shares.rs:351-357`). Runtime validation allows an `http://localhost` public URL outside dev mode because localhost is treated as a local exception (`vault/server/src/auth.rs:931-944`, `vault/server/src/auth.rs:969-985`).

Impact: a production Docker Compose deployment can set `VAULT_AUTH_MODE=oidc`, configure the issuer/client/secret, put nginx with TLS in front, and still generate `redirect_uri=http://localhost:8000/auth/callback` unless the operator explicitly overrides `VAULT_PUBLIC_URL` or `VAULT_OIDC_REDIRECT_URI`. It can also issue share URLs such as `http://localhost:8000/s/...` instead of the public HTTPS origin. This breaks OIDC login and user-facing links, and can be hard to diagnose because correct nginx forwarded headers are ignored once the compose default is present. It is not a database corruption risk, but it is a high-probability production deployment failure for the exact nginx/TLS/OIDC topology.

Recommendation: do not default `VAULT_PUBLIC_URL` to localhost in production Compose; leave it unset so forwarded derivation works for callbacks and relative share links can stay relative, or require an explicit public HTTPS URL for production. Fail OIDC startup when `VAULT_PUBLIC_URL`/`VAULT_OIDC_REDIRECT_URI` is localhost while `VAULT_AUTH_MODE=oidc` and `VAULT_DEV_MODE=0`. Add startup warning/validation tests for the compose-default OIDC and share-link cases.

### Auth Deployment - Non-origin public URLs are accepted even though the app is root-mounted

Runtime validation only checks that `VAULT_PUBLIC_URL` and the derived OIDC redirect origin are absolute HTTP(S) URLs with an acceptable scheme/host; it does not require an origin-only value or reject a path/query/fragment such as `https://example.com/vault` (`vault/server/src/auth.rs:931-985`, `vault/server/src/auth.rs:1001-1016`). The OIDC redirect builder then appends `/auth/callback` to the full configured value, and share-link creation similarly appends `/s/{code}` (`vault/server/src/http.rs:2681-2696`, `vault/server/src/shares.rs:351-357`). But the server registers root-mounted routes like `/auth/callback`, `/s/{code}`, `/static/{*path}`, and `/api/...` (`vault/server/src/http.rs:130-186`, `vault/server/src/http.rs:216-256`). The generated HTML also requires manifest paths to start with `/static/dist/`, and the first-party client uses absolute `/api/...`, `/folders`, `/documents/...`, `/health`, and `/api/events/stream` URLs (`vault/server/src/assets.rs:148-176`, `vault/client/src/lib/useVaultResources.js:223-314`, `vault/client/src/lib/useVaultResources.js:430-479`, `vault/client/src/lib/transferClient.js:108-384`).

Impact: an nginx deployment that publishes Vault under a subpath can pass startup validation and generate OIDC/share URLs containing that subpath, while the actual app shell, static assets, API calls, SSE stream, and callback routes remain root-relative unless nginx rewrites every path perfectly. The likely failure mode is an OIDC redirect mismatch, broken static/assets/API requests, or share links that work differently from the live app. This is not a data-corruption path, but it is a production deployment footgun because the app accepts a configuration shape it does not consistently support.

Recommendation: either explicitly support a base path throughout the router, asset manifest, cookie paths, frontend API URLs, SSE URL, download URLs, OIDC callback, and share URLs, or reject `VAULT_PUBLIC_URL`/`VAULT_OIDC_REDIRECT_URI` values with anything beyond scheme/host/optional port at startup. Document root-domain/subdomain deployment as the supported nginx topology until base-path support is implemented.

### Auth / Redirects - OIDC and logout return paths allow browser-normalized backslash redirects

OIDC login and logout both accept a user-controlled `rd` return path. The login route sanitizes `rd` with `safe_redirect` and stores the result in the signed OIDC state cookie, then the callback redirects to the stored value after successful token exchange (`vault/server/src/http.rs:438-462`, `vault/server/src/http.rs:520-530`, `vault/server/src/oidc.rs:103-114`). Logout uses the same helper directly in the `Location` header (`vault/server/src/http.rs:533-542`). Both helper implementations only require the string to start with `/` and not start with `//` (`vault/server/src/http.rs:2732-2737`, `vault/server/src/oidc.rs:421-426`). The route test covers an absolute `https://evil.example.com` return path, but I did not find coverage for decoded backslash forms (`vault/server/tests/routes_oidc.rs:830-858`).

Impact: a crafted return path such as `/%5Cevil.example.com` can be decoded to `/\evil.example.com`, pass the current helper, and be emitted as a `Location` header. Browser URL parsers commonly normalize backslashes in special URLs as slashes, so that header can become a network-path redirect to `https://evil.example.com/`. This does not leak the server-side OIDC code exchange or Vault session cookie to the attacker domain, but it gives the production OIDC login/logout flow an open-redirect primitive useful for phishing and confusing post-auth navigation.

Recommendation: use one shared redirect sanitizer that parses the candidate as a relative URL and rejects backslashes, control characters, encoded or decoded network-path forms, non-HTTP path oddities, and any absolute origin. Normalize accepted paths before storing them in OIDC state or writing a `Location` header. Add regression tests for `rd=//evil.example.com`, `rd=https://evil.example.com`, `rd=/%5Cevil.example.com`, `rd=/\evil.example.com`, and encoded variants on both login/callback and logout.

### Auth / OIDC Availability - Userinfo is a hard login dependency whenever discovery advertises it

The callback validates the state cookie, exchanges the code, verifies the ID token, and then calls `userinfo(...)` whenever the provider discovery document has `userinfo_endpoint` and the token response has a non-empty `access_token` (`vault/server/src/oidc.rs:103-132`, `vault/server/src/oidc.rs:279-309`). A non-2xx userinfo response, timeout, JSON error, or provider network failure bubbles up as an OIDC callback failure through `json_response`/`ProviderRequest` (`vault/server/src/oidc.rs:279-328`). The ID token is already verified for issuer, audience, nonce, and signature before this call, and subject/email/name/groups can be read from the merged identity map after the optional userinfo extension (`vault/server/src/oidc.rs:238-277`, `vault/server/src/oidc.rs:129-136`). Tests cover successful userinfo merge and subject mismatch rejection, but I did not find a mode or test that allows login to proceed from sufficient ID-token claims when the userinfo endpoint is temporarily unavailable (`vault/server/tests/routes_oidc.rs:861-924`, `vault/server/tests/routes_oidc.rs:1080-1110`).

Impact: a partial IdP outage can lock every OIDC user out of Vault even when authorization code exchange and ID-token validation are healthy. That is especially risky for a production deployment behind nginx/TLS because the app has no local break-glass login path in OIDC mode unless header/dev auth is reconfigured and restarted. This does not corrupt data, but it turns an optional OIDC endpoint into an availability dependency for all browser sessions that need to refresh or log in.

Recommendation: make userinfo optional when the verified ID token already contains the configured subject/email/name/group claims needed for the deployment, or add a `VAULT_OIDC_REQUIRE_USERINFO`/claim-source setting. Preserve the subject mismatch protection when userinfo succeeds, but treat timeout/provider errors as degraded login only when policy allows ID-token-only identity. Add tests for userinfo 500/timeout/invalid JSON with ID-token claims present and absent.

### Auth Deployment - Session and OIDC state cookie names are not required to be distinct

Runtime validation checks `VAULT_SESSION_COOKIE_NAME` and `VAULT_OIDC_STATE_COOKIE_NAME` for allowed characters, but it does not reject equal names (`vault/server/src/auth.rs:250-288`). The OIDC login route writes the signed state payload to `oidc_state_cookie_name` (`vault/server/src/http.rs:451-482`). The callback then emits two `Set-Cookie` headers in one redirect: first the new session under `session_cookie_name`, then a deletion for `oidc_state_cookie_name` (`vault/server/src/http.rs:518-529`, `vault/server/src/http.rs:2764-2769`). The custom-cookie test only covers distinct names (`vault/server/tests/routes_oidc.rs:563-594`).

Impact: if an operator sets both names to the same value, `/login` overwrites any current session with an OIDC state cookie, and `/auth/callback` can delete the just-created session cookie in the same response. The result is an OIDC login loop or intermittent unauthenticated state after apparently successful callbacks. This is a configuration footgun rather than data corruption, but it is a production availability failure in the exact OIDC cookie deployment path.

Recommendation: reject equal session and OIDC state cookie names during startup validation and add a regression test for the collision case. Logout can continue deleting both names, but login/callback should require a non-overlapping cookie namespace.

### Auth Deployment - Secure session cookies can be explicitly disabled in production

Runtime validation accepts `VAULT_SESSION_COOKIE_SECURE=false`/`0`/`off` in all modes (`vault/server/src/auth.rs:250-256`, `vault/server/src/auth.rs:910-915`). The cookie builder then gives that explicit false value priority over both `VAULT_PUBLIC_URL=https://...` and `X-Forwarded-Proto: https`, omitting the `Secure` attribute from OIDC state and session cookies (`vault/server/src/http.rs:2719-2729`, `vault/server/src/http.rs:2751-2761`). Route tests assert this override for both login state cookies and callback session cookies even with an HTTPS public URL or forwarded HTTPS header (`vault/server/tests/routes_oidc.rs:663-710`). README correctly recommends leaving `VAULT_SESSION_COOKIE_SECURE=auto` for TLS termination (`README.md:21-23`), but startup does not enforce that recommendation.

Impact: a production OIDC deployment can accidentally run with browser session cookies that are not marked `Secure`. TLS still protects normal HTTPS requests, but the cookies become eligible for cleartext HTTP requests to the same host if HSTS is absent, not preloaded, mis-scoped, or bypassed during first contact. This is not a database consistency problem, but it weakens the core OIDC session boundary.

Recommendation: outside development, reject explicit false when `VAULT_AUTH_MODE=oidc` or when `VAULT_PUBLIC_URL`/forwarded deployment guidance indicates HTTPS. Keep `auto` and explicit true; reserve explicit false for local development with `VAULT_DEV_MODE=1`. Add startup tests for production OIDC plus `VAULT_SESSION_COOKIE_SECURE=false`.

### Auth Deployment - Header-auth is the Compose default and trusts request identity headers directly

The production compose file defaults to `VAULT_AUTH_MODE: ${VAULT_AUTH_MODE:-headers}` (`docker-compose.yml:29`), and `.env.example` also defaults to `VAULT_AUTH_MODE=headers` while describing OIDC as an explicit alternative (`.env.example:24-39`). In header mode, `current_user` calls `header_identity` (`vault/server/src/http.rs:2644-2657`), and `header_identity` accepts `Remote-User`, `Remote-Name`, `Remote-Email`, and `Remote-Groups` directly from the request headers, syncing groups and admin status from `Remote-Groups` (`vault/server/src/auth.rs:339-380`). The tests intentionally assert that a `Remote-Groups: vault-admin` header yields admin context (`vault/server/tests/auth.rs:72-101`).

Impact: if this container is exposed directly, or if nginx forwards client-supplied `Remote-*` headers instead of clearing and overwriting them, header mode is an authentication bypass. For the intended OIDC deployment this is avoidable by setting `VAULT_AUTH_MODE=oidc`, but the provided Compose default is still dangerous for a serious production environment if copied unchanged.

Recommendation: for OIDC production, make `VAULT_AUTH_MODE=oidc` explicit and fail deployment if the OIDC variables are missing. If header auth remains supported, document exact nginx header stripping/setting rules and consider a separate explicit `VAULT_TRUST_HEADER_AUTH=1` style guard so the production default is not direct header trust.

### Authorization - Case-variant group rows are treated as the same group by access checks

Group names are not canonicalized consistently. Header/OIDC group claims are lowercased by `split_groups`, and `sync_vault_groups` looks up or creates `vault_groups` with an exact `WHERE name = ?` match (`vault/server/src/auth.rs:618-677`). Admin group creation preserves case after whitespace normalization, and its case-insensitive duplicate check is a separate pre-insert `SELECT lower(name) = ?` rather than a database constraint (`vault/server/src/admin.rs:241-252`, `vault/server/src/admin.rs:629-637`). The schema's `uq_vault_groups_name` unique constraint is case-sensitive SQLite text uniqueness (`vault/server/src/db/schema.rs:54-60`).

Access checks then compare by lowercased names, not group ids. `folder_access_level` loads permission rows by `folder_permissions.group_id`, joins the row's group name, and grants access when the user's group-name set contains `row.group_name.trim().to_ascii_lowercase()` (`vault/server/src/folders.rs:758-793`). A membership in group row `team` can therefore satisfy a permission assigned to a different group row `Team`, even though the admin directory and permission tables show separate ids.

Impact: an admin can create a mixed-case group, then a later header/OIDC login with the lowercased name can create a second row. Folder permissions, admin membership views, and group deletion guards operate by row id, while runtime access collapses those rows by lowercased name. That can produce surprising read/write access, make permission reviews misleading, and make revocation/deletion difficult because removing permissions or memberships from one case-variant row may not remove the effective access path.

Recommendation: choose one group identity model and enforce it everywhere. Prefer storing a canonical lowercased group key with a unique index and display name separately, then make memberships and folder permissions compare ids through that canonical key. Add a migration that merges case-variant groups safely, and tests for admin-created mixed-case groups followed by OIDC/header sync of the lowercased name.

### Reverse Proxy / OIDC - Forwarded headers and audit IP are trusted without an implemented trusted-proxy boundary

The app uses `X-Forwarded-Proto` to decide whether a request is public HTTPS and whether to set HSTS/cookie `Secure` (`vault/server/src/http.rs:2527-2530`, `vault/server/src/http.rs:2713-2729`). When no explicit `VAULT_PUBLIC_URL` or `VAULT_OIDC_REDIRECT_URI` is configured, it uses `X-Forwarded-Host` with `Host` fallback plus `X-Forwarded-Proto` to build the OIDC callback URL (`vault/server/src/http.rs:2681-2703`). Audit metadata also takes the first `X-Forwarded-For` value directly (`vault/server/src/http.rs:2154-2166`).

`.env.example` tells operators to set `FORWARDED_ALLOW_IPS` (`.env.example:30-31`), but I did not find server code that reads it, and `docker-compose.yml` does not pass it through. The production compose test asserts compose does not contain `FORWARDED_ALLOW_IPS` (`vault/server/tests/config.rs:325-327`).

Impact: if the container is reachable directly, or nginx forwards client-supplied forwarded/host headers instead of overwriting them, clients can influence HTTPS detection, audit IPs, and generated OIDC redirect URLs when `VAULT_PUBLIC_URL` is unset. This is mainly deployment-boundary/auth hardening, not a standalone database corruption issue.

Recommendation: require `VAULT_PUBLIC_URL=https://...` in production and document nginx header clearing/setting, or implement trusted-proxy CIDR enforcement before honoring forwarded headers. Bind the compose service only to loopback/private networks unless a trusted proxy is in front.

### Operations - Docker health/readiness does not exercise SQLite integrity or object storage

The Docker healthcheck calls `/health` (`Dockerfile:50-51`), and `/health` always returns `"ok"` (`vault/server/src/http.rs:283-285`). `/api/health` runs `SELECT 1`, but still returns HTTP 200 with an `ok` boolean and does not test storage (`vault/server/src/http.rs:561-569`). Startup calls `storage.ensure()` (`vault/server/src/main.rs:21-31`), but S3/R2 `ensure()` is a no-op (`vault/server/src/storage.rs:850-852`). Startup validates the live schema shape before seeding roots (`vault/server/src/db/mod.rs:70-85`, `vault/server/src/db/mod.rs:358-425`), but I did not find an integrity probe such as `PRAGMA integrity_check`, `PRAGMA quick_check`, or `PRAGMA foreign_key_check`, and the only storage reconciliation route is a dev-only admin debug endpoint with `apply=false` (`vault/server/src/http.rs:873-883`).

Impact: Docker/nginx can keep routing traffic to a container whose SQLite pool, `/data` volume, S3/R2 bucket/credentials, foreign-key graph, or referenced object set are unhealthy. S3/R2 misconfiguration may not be detected until the first upload/export operation, and local missing/corrupt referenced objects may not be detected until a user starts a download or export.

Recommendation: add a readiness endpoint that returns non-2xx when SQLite cannot query quickly, `integrity_check`/`foreign_key_check` fails, local storage is not writable/readable, referenced local objects are missing/corrupt, or remote storage cannot perform a low-cost bucket/head/probe check. Keep `/health` as liveness if needed, but point the Docker healthcheck at readiness and surface degraded storage/DB integrity in operations docs.

### Deployment Release Control - Compose deploys a pinned published image, not this checkout

The default Compose service uses `image: ${VAULT_IMAGE:-ghcr.io/willjallen/vault:v1.0.0}` and has no `build:` section (`docker-compose.yml:1-7`). `.env.example` also sets `VAULT_IMAGE=ghcr.io/willjallen/vault:v1.0.0` (`.env.example:1-8`). The README states that this is intentional and tells operators to set `VAULT_IMAGE` only when upgrading, while the local build path is documented under the dev override (`README.md:5-19`). The dev override does add `build:`, but it also enables dev-mode settings and is explicitly not for production (`docker-compose.dev.yml:1-8`, `README.md:15-23`).

Impact: an operator can audit, patch, or test this working tree and still deploy the pinned registry image by running the documented production `docker compose up -d`. If the pinned image is older than the audited source, missing a migration, or built from different frontend/server assets, production behavior can diverge from the findings and from local validation. This is not direct database corruption, but it is a release-control footgun for a serious deployment because the artifact actually handling OIDC, migrations, transfer recovery, and storage cleanup may not be the artifact that was reviewed.

Recommendation: document an explicit production build/pin flow that ties the deployed image digest to the audited commit. For example, build a production image from this checkout, push it under an immutable tag/digest, set `VAULT_IMAGE` to that digest, and record the source commit in release notes. Avoid using the dev Compose override as the production build path unless it is split into a production-safe build override with no dev auth/mode settings.

### Upgrade / Availability - Some additive schema paths can fail their own validation

Startup runs known additive migrations before strict schema validation (`vault/server/src/db/mod.rs:96-123`, `vault/server/src/db/mod.rs:358-426`). The missing-table creation path only creates `vault_settings`, `upload_sessions`, `upload_parts`, `export_jobs`, and `export_artifacts` when `vault_users` already exists; it does not create missing `share_links` or `state_events` tables even though validation requires every table in the current schema (`vault/server/src/db/mod.rs:100-110`, `vault/server/src/db/schema.rs:284-313`). Separately, the share-link migration adds missing columns to an existing `share_links` table with loose definitions: `target_type TEXT`, `document_id INTEGER`, `folder_id INTEGER`, and `created_by_user_id INTEGER` (`vault/server/src/db/mod.rs:227-255`). It then backfills target fields from legacy `item_type`/`item_id` values (`vault/server/src/db/mod.rs:257-291`).

The expected current schema is stricter. `share_links.target_type` is `TEXT NOT NULL`, `document_id` and `folder_id` have foreign keys with `ON DELETE CASCADE`, and `created_by_user_id` has a foreign key with `ON DELETE SET NULL` (`vault/server/src/db/schema.rs:292-308`). Validation compares live column metadata and foreign keys to that expected schema and rejects mismatches (`vault/server/src/db/mod.rs:367-405`). SQLite cannot add those table-level constraints with the current `ALTER TABLE ... ADD COLUMN` definitions, so a legacy table that takes this migration path can still be refused at startup as incompatible.

Impact: an upgrade can leave the database unmodified-but-unstartable until an operator performs a manual table creation or table rebuild migration. Existing share links and other metadata are preserved, which is good for data safety, but the container stays down after the attempted upgrade. The current tests cover several validation failures and some additive column/table cases, but I did not find a regression that exercises a missing `share_links`/`state_events` table or a legacy `share_links` table through this migration path.

Recommendation: replace the share-link additive path with an explicit table rebuild migration that creates the current constrained table, copies valid rows, handles invalid legacy targets deliberately, recreates indexes, and then passes validation. Add a startup regression test using a legacy `share_links` schema with `item_type`/`item_id` rows.

### Storage / S3-R2 - Remote read and HEAD failures are collapsed into missing-object semantics

The S3/R2 backend maps any `get_object` request failure in `read_bytes` and `read_range` to `StorageError::NotFound` (`vault/server/src/storage.rs:918-960`). The HTTP download mapper then returns 404 `"Blob missing from storage"` for that error (`vault/server/src/http.rs:2422-2427`, `vault/server/src/http.rs:3472-3488`). Separately, the dedupe probe `remote_object_exists` returns a plain boolean and treats every `head_object` error as "does not exist" (`vault/server/src/storage.rs:981-990`), so write paths cannot distinguish absent objects from permission, credential, endpoint, throttling, or transient provider failures before attempting a PUT (`vault/server/src/storage.rs:854-896`).

Impact: a temporary S3/R2 outage or credential regression can look exactly like permanent object loss to users and operators. That can hide the difference between "metadata points at missing bytes" and "the storage backend is unavailable," delaying the right incident response and making data-loss triage harder. During writes, failed HEAD probes also bypass the intended existing-object decision point, so remote storage behavior depends on the later PUT outcome rather than an explicit storage-health/error path.

Recommendation: preserve remote provider errors separately from true 404/no-such-key responses. Map transient/auth/provider failures to 5xx storage-unavailable responses, log enough backend detail for operators, and make `remote_object_exists` return `Result<bool, StorageError>` so write paths fail fast on ambiguous HEAD failures instead of treating them as absence.

### Storage / S3-R2 - Remote object operations have no Vault-level timeout controls

OIDC provider calls have an explicit timeout setting: `VAULT_OIDC_HTTP_TIMEOUT_SECONDS` is parsed into auth settings and every OIDC reqwest client is built with that duration (`vault/server/src/auth.rs:220-234`, `vault/server/src/oidc.rs:340-345`). The S3/R2 storage client does not expose a comparable Vault setting or apply an operation timeout in its builder; `S3CompatibleBlobStorage::from_settings` configures region, credentials, endpoint, and path style, then builds the AWS client (`vault/server/src/storage.rs:776-831`). All remote object operations are awaited directly: HEAD-before-put, PUT, GET/range body collection, and DELETE (`vault/server/src/storage.rs:854-978`).

Those awaits sit on important request and maintenance paths. Upload completion waits on remote staging/put before committing document/version state (`vault/server/src/uploads.rs:862-957`). Downloads and export ZIP workers wait on remote reads before responding or finishing artifacts (`vault/server/src/http.rs:2274-2315`, `vault/server/src/exports.rs:1218-1230`, `vault/server/src/exports.rs:1322-1331`). Startup transfer sweep/recovery and the periodic sweeper await remote deletes after committing metadata cleanup (`vault/server/src/main.rs:31-44`, `vault/server/src/main.rs:55-74`, `vault/server/src/transfers.rs:99-124`, `vault/server/src/transfers.rs:170-250`).

Impact: with S3/R2 or a compatible endpoint, a provider/network stall can tie up upload completions, downloads, export workers, startup recovery, or the single periodic cleanup loop for as long as the underlying SDK/default transport permits. That does not directly corrupt SQLite, but it can make the container appear hung, keep Docker readiness green on `/health`, leave transfer rows in intermediate states until timeout/restart, and stop future TTL/transfer cleanup while the sweeper is stuck in one storage call.

Recommendation: add explicit remote-storage operation timeout and retry settings, apply them to HEAD/GET/PUT/DELETE and body collection, and map timeout/retry exhaustion to a storage-degraded response instead of missing-object semantics. Keep maintenance loops from blocking forever on one object by bounding each delete/read call and continuing with observable retry state. Add integration tests with a stalled mock S3 endpoint for upload completion, range download, export, startup recovery, and transfer sweep.

### Storage / Local - Multipart part cleanup can leave hidden retained bytes

Local `delete_object` handles multipart objects by loading the manifest, deleting the manifest path if present, and then best-effort deleting each referenced part while ignoring every part deletion error (`vault/server/src/storage.rs:425-444`). The local object lister intentionally hides multipart part keys (`vault/server/src/storage.rs:1228-1257`), and storage reconciliation builds its local view from that lister (`vault/server/src/reconciliation.rs:98-109`, `vault/server/src/reconciliation.rs:195-201`).

Impact: if deleting a multipart object removes the manifest but fails to remove one or more part files, those bytes become invisible to normal object listing and reconciliation. Deleted export artifacts, failed upload cleanup, admin reconciliation, or manual maintenance can therefore leave retained document/export contents under `/data/objects/multipart/.../parts` with no manifest, no database location, and no later retry path. This is a data-retention and disk-exhaustion risk that can make readiness look clean while the object volume keeps growing.

Recommendation: make multipart deletion durable and observable. Delete parts before the manifest or keep a retryable tombstone/cleanup record until every part deletion succeeds; log and surface part cleanup failures; and add a reconciliation pass that scans multipart part directories for orphaned parts whose manifest is missing or whose manifest is no longer referenced.

### Export Integrity - ZIP entry de-duplication can still emit duplicate names

Export ZIP entry names are derived from `download.document_path` after sanitization (`vault/server/src/exports.rs:1158-1166`, `vault/server/src/exports.rs:2134-2167`). The sanitizer removes characters such as `:`, `*`, `?`, quotes, angle brackets, and `|`, while normal folder and document names only reject path separators, `.`/`..`, and control characters (`vault/server/src/folders.rs:154-171`, `vault/server/src/documents.rs:1588-1602`). When a sanitized name collides, the code prefixes the document id once, but it does not check whether that fallback name was already used (`vault/server/src/exports.rs:1160-1164`).

Impact: a folder export can contain duplicate central-directory names even though the source documents have distinct Vault paths. Many unzip tools present or extract only one entry for a duplicated name, often overwriting the earlier file on disk. That makes the export artifact a lossy representation of the selected files, which is especially risky if operators use folder export as an ad hoc backup or migration path.

Recommendation: allocate ZIP entry names in a loop until `written_names.insert(...)` succeeds. Include the document id plus a collision counter or stable suffix before the extension, and add tests for sanitizer collisions plus fallback-name collisions.

### Deployment Topology - Multi-replica operation is neither enforced nor documented as unsupported

The shipped compose file starts one app container, and the README describes a standard deployment with one persistent `/data` volume, but I did not find an explicit production warning or runtime guard that Vault is a single-container/single-writer service (`README.md:5-13`, `docker-compose.yml:1-78`). The server holds several coordination primitives only in process memory: `AppState` owns a per-process export execution context, upload hash coordinator, and upload part lock set (`vault/server/src/http.rs:86-113`). Duplicate upload part writes are serialized with `UploadPartLock` over an in-memory `HashSet` keyed by session/part (`vault/server/src/http.rs:1334-1400`), and pre-completion upload hashing keeps per-session state in an in-memory `HashMap` and schedules process-local tasks (`vault/server/src/uploads.rs:267-388`).

Exports are partially persisted in SQLite, but execution concurrency is still process-local: `ExportExecutionContext` uses a per-process semaphore (`vault/server/src/exports.rs:138-166`), queued jobs are spawned into the current process (`vault/server/src/exports.rs:622-735`), and only the later claim update prevents duplicate execution (`vault/server/src/exports.rs:974-1008`). State-event wakeups are also process-local broadcasts (`vault/server/src/state_events.rs:91-137`). The performance notes explicitly say multiple workers require sticky routing or stronger cross-process coordination, but that warning lives in an internal problem document rather than the production deployment path (`docs/PROBLEM.md:424-439`).

Impact: an operator scaling the container behind nginx, a load balancer, or an orchestrator can bypass upload part serialization, lose upload preverification locality, multiply export worker concurrency beyond `VAULT_EXPORT_WORKERS`, and run multiple startup/sweeper/recovery loops against the same SQLite/filesystem state. SQLite constraints and atomic file promotion prevent some direct corruption, but this is an unsupported topology with higher odds of transfer conflicts, stale-state overwrites, delayed UI invalidation, and resource exhaustion.

Recommendation: make the singleton requirement explicit in README/Compose/OIDC-nginx docs and add a startup guard such as a lock file/flock under `/data` or a DB-backed lease so accidental second replicas fail closed. If horizontal scaling is a goal, move coordination into SQLite/file locks or an external queue, add sticky routing for upload sessions, and test uploads/exports/recovery with requests spread across workers.

### Operations - State invalidation grows indefinitely, while audit-event retention is undefined

`state_events` is used as an SSE invalidation feed, but it is persisted in SQLite and only has an insert/query API: table creation and index (`vault/server/src/db/schema.rs:284-291`), inserts (`vault/server/src/state_events.rs:69-88`, `vault/server/src/documents.rs:1511-1525`, `vault/server/src/uploads.rs:1987-2001`, `vault/server/src/folders.rs:1125-1140`), and reads after a client's last event id (`vault/server/src/state_events.rs:41-67`). I did not find a production prune/delete path for `state_events`; startup and periodic sweeps clean TTL documents/transfers, not state events (`vault/server/src/main.rs:31-45`, `vault/server/src/main.rs:55-74`).

High-volume actions also write document history. For example, every download writes a `document_events` row and a `state_events` row in `record_download_event` (`vault/server/src/documents.rs:560-586`), and export completion records download-style history entries for exported files (`vault/server/src/exports.rs:1544-1566`). Folder metadata/permission/retention changes write `folder_events` history (`vault/server/src/db/schema.rs:25-35`, `vault/server/src/folders.rs:1096-1118`). Those audit tables do not have an explicit prune/export policy, and they are not clearly permanent either: `document_events` and `folder_events` both cascade when their parent document or folder is deleted (`vault/server/src/db/schema.rs:25-35`, `vault/server/src/db/schema.rs:178-190`). Document/folder history may be intentional per-object history rather than immutable audit history, but the product does not currently define the retention boundary. The SSE feed is operational invalidation state and currently grows indefinitely with normal UI/API traffic.

Impact: over time, a busy production instance can grow `vault.db` with ephemeral state events and high-volume download/export history. That increases backup/restore time, WAL/checkpoint work, disk usage, and the amount of data retained beyond its operational usefulness. It is not an immediate corruption vector, but it can become an availability and maintenance problem in long-lived deployments.

Recommendation: define retention separately for audit history and ephemeral state invalidation. Prune `state_events` by age or by keeping the latest N IDs after all connected clients can safely resync from a fresh bootstrap. If `document_events` and `folder_events` are intended to be permanent audit history, move them away from parent-row cascades and document the storage growth expectation; otherwise document that they are per-object history and add configurable retention/export for old rows.

### Fault Reporting - Bulk/admin/preference mutations can commit before their state event is written

Bulk document/folder routes execute each item mutation in its own committed transaction, then write one aggregate `state_events` row afterward. Lock, unlock, move, archive, restore, delete-forever, and rename routes all call the mutating helper first and then call `record_document_batch_state` or `record_document_deleted_state` outside those item transactions (`vault/server/src/http.rs:1525-1860`). Those state helpers are separate SQLite writes (`vault/server/src/documents.rs:948-975`).

Admin user/group routes have the same reporting shape. `update_user`, `create_group`, `update_group`, `delete_group`, `add_group_member`, and `remove_group_member` commit the user/group mutation first, then call `record_admin_change`, which inserts a `state_events` row through a separate database statement (`vault/server/src/admin.rs:208-376`, `vault/server/src/admin.rs:660-676`). Site settings are a useful contrast: `update_admin_site_settings` writes the settings and `admin.settings.updated` event inside one transaction (`vault/server/src/site_settings.rs:38-70`).

Preferences are another smaller instance of the same pattern. `update_preferences_for_user` updates `vault_users.preferences` and commits its transaction before returning (`vault/server/src/preferences.rs:45-76`); the HTTP handler then writes `preferences.update` as a separate state event and returns an error if that later insert fails (`vault/server/src/http.rs:1032-1045`). The preference patch validator is strict about known keys and value types, so the issue is not arbitrary preference shape corruption; it is a committed side effect that can be reported to the caller as a failed request.

Impact: if SQLite is full, locked, or otherwise errors during the state-event insert, the API can return a top-level 500 after some or all requested mutations already committed. Clients may not receive the SSE invalidation, and a retry can see a different world: files may already be moved, archived, deleted, locked, or restored, or users/groups may already be changed. That should not corrupt SQLite because each domain mutation is atomic, but it creates exactly the kind of misleading "failed" response with persistent side effects that makes production support and user recovery difficult.

Recommendation: either include the state event in the same transaction as each mutation, make the notification best-effort after returning the committed result, or expose a clear "mutation committed but notification failed" response. Add tests that inject a state-event insert failure after successful document, admin, and preference mutations and assert the response/notification semantics are explicit.

### State Sync - Initial SSE subscription can miss mutations between bootstrap and stream start

The browser renders from a bootstrap payload that includes auth/user/preferences/settings/current folder, contents, sidebar, and edits, but it does not include a durable `state_events` watermark (`vault/server/src/views.rs:64-74`, `vault/server/src/views.rs:367-383`). The client then opens `new EventSource("/api/events/stream")` without a `Last-Event-ID` header (`vault/client/src/lib/useVaultResources.js:430-479`). On the server, a stream without `Last-Event-ID` starts at `latest_state_event_id(pool)` and only replays later rows (`vault/server/src/http.rs:2182-2189`, `vault/server/src/http.rs:2231-2246`). The test suite asserts this behavior: old events are intentionally skipped when no last event id is supplied (`vault/server/tests/routes_events.rs:164-186`).

Impact: a mutation that commits after the bootstrap/initial-state queries but before the SSE request establishes its starting id can be permanently missed by that browser tab. The next unrelated state event may refresh the stale resource, and server-side authorization/path checks still protect writes, so this is not a database corruption path. It can still leave users looking at stale contents/sidebar/details after another user's change, increasing failed actions, confusing transfer workflows, and support reports that the live UI did not update.

Recommendation: include the current `state_events` id in the bootstrap/initial-state response after all data reads, and have the client connect with that id or an equivalent query parameter on first SSE subscription. Alternatively, force an initial refresh after the stream opens. Add a test that inserts a state event between bootstrap construction and SSE connection and proves the client receives or compensates for it.

### State Sync - One malformed state event row can wedge SSE replay

`state_events.resources` is stored as plain text with no JSON validity constraint (`vault/server/src/db/schema.rs:284-291`). The stream replay helper selects rows after the client's last id and converts the whole batch into `StateEventRecord`s; parsing a single malformed `resources` value returns an error for the entire batch (`vault/server/src/state_events.rs:49-67`, `vault/server/src/state_events.rs:100-129`). The SSE loop logs that error but does not skip, quarantine, or advance past the bad row; it then waits for another notification and re-runs the same failing query from the same `last_id` (`vault/server/src/http.rs:2231-2246`). Reconnecting browsers send `Last-Event-ID`, so any client whose last delivered id is before the malformed row will keep requesting replay from before that row (`vault/server/src/http.rs:2182-2190`, `vault/client/src/lib/useVaultResources.js:430-479`).

Impact: an application bug, bad manual repair, partial migration, or SQLite corruption in one non-canonical invalidation row can permanently stop live updates for affected connected clients even if later state rows are valid. Users can keep operating on stale contents/sidebar/detail/preference state until a hard refresh or until the bad row is repaired, while new clients that start at the latest id may appear healthy. That split makes production incidents hard to diagnose.

Recommendation: validate state-event rows at insert time with a JSON check or normalized child table, and make SSE replay fault-isolating. If a historical row is malformed, emit a full-refresh event, advance past or quarantine the bad row with an operator-visible alert, and continue delivering later events. Add a regression test with one malformed `state_events.resources` row followed by a valid row to prove the stream does not wedge.

### SQLite Contention - Segmented downloads write audit and state rows for every range

Every document download route reads the requested bytes into a response and only then records a download audit event plus a `document.download` state event before returning the response (`vault/server/src/http.rs:1206-1212`, `vault/server/src/http.rs:1471-1481`, `vault/server/src/http.rs:1512-1521`, `vault/server/src/documents.rs:560-587`). Tests assert that even a partial range request records both rows (`vault/server/tests/routes_documents.rs:1083-1157`). The first-party browser downloader intentionally starts with `Range: bytes=0-0` and, for large files, switches to concurrent segmented range requests (`vault/client/src/lib/transferClient.js:729-849`, `vault/client/src/lib/transferClient.js:852-910`).

Impact: a large first-party download can create one `document_events` row and one `state_events` row per segment, not per user-visible download. With 64 MiB segments, a 5 GiB file produces about 80 write transactions and 80 SSE invalidations while it is being read. If SQLite is locked, full, or slow after the bytes are already read from storage, the route can fail the download response even though the requested object data was available. This is not metadata corruption, but it turns read-only transfer throughput into sustained SQLite write pressure and can make downloads appear hung or flaky during unrelated writer contention.

Recommendation: treat download audit as a best-effort or coalesced operation on the hot path. Record one logical download per user action/job instead of one per range, skip audit/state writes for range probes, or batch/throttle range events. If audit is mandatory, write the audit event before expensive storage reads or return a clear "audit failed before download" error rather than discarding already-read bytes. Add tests for segmented downloads proving bounded audit/state writes.

### Data Lifecycle - Expired locked documents can keep retention permanently pending

Document locks have no expiry column or stale-lock policy; an active lock remains active until an owner/admin unlocks it, check-in releases it, the document is archived/deleted by an allowed user, or the document row is deleted through cascade (`vault/server/src/db/schema.rs:142-157`, `vault/server/src/documents.rs:285-322`, `vault/server/src/uploads.rs:1040-1091`). Admin user disablement only updates `vault_users.is_admin`/`is_active` and records the admin event, so it does not release locks held by the disabled account (`vault/server/src/admin.rs:208-236`). The retention sweeper intentionally selects expired documents, queries active locks, and skips any locked document while leaving `expires_at` and `expiry_action` unchanged (`vault/server/src/documents.rs:352-428`). The retention test asserts this exact behavior: an expired delete-TTL document with an active lock is returned in `result.skipped`, remains in `documents`, and still has its expiry fields set (`vault/server/tests/retention.rs:232-272`).

Impact: delete/archive TTL policies are not guaranteed to converge in production if a user checks out a document and leaves, loses access, or their session/account is disabled before release. Every sweep will continue to skip the row, so data meant to be deleted or archived can remain indefinitely until an admin notices and unlocks it. Because the sweeper orders expired rows by `expires_at`/`id` and processes a fixed limit, a large set of old locked rows can also monopolize the sweep window and delay newer unlocked expirations behind them (`vault/server/src/documents.rs:1106-1122`). This is safer than deleting locked work, but without stale-lock visibility/escalation it creates a persistent faulty lifecycle state and retention compliance risk.

Recommendation: add a stale-lock policy and operator visibility before enforcing retention against locked documents. Options include an admin stale-lock dashboard, configurable lock max age with notification/escalation, a retention report for repeatedly skipped documents, or an admin-only forced unlock flow that records audit history. Add tests for repeated sweeps over a stale lock and for the chosen admin remediation path.

### Edit Workflow - Abandoned document locks have no expiry or broad recovery surface

`document_locks` stores `locked_at`, `is_active`, release metadata, and a unique active lock per document, but no expiry/deadline/heartbeat field (`vault/server/src/db/schema.rs:142-157`). `lock_document` and checkout-created locks only accept the existing lock when the caller is the lock owner or an admin; otherwise they return `DocumentLockedByOtherUser` (`vault/server/src/documents.rs:239-283`, `vault/server/src/documents.rs:590-634`, `vault/server/src/documents.rs:1336-1345`). The main "my edits" payload loads all active locks but only returns rows where `lock.locked_by == user.id`, so it is not an operator-wide stale-lock queue (`vault/server/src/views.rs:437-470`). I did not find a lock heartbeat, max-age sweeper, or documented startup/admin maintenance flow that releases abandoned locks automatically.

Impact: a user can check out or lock a document, then leave the company, lose OIDC group access, close the browser, or abandon the edit. Normal writers stay blocked indefinitely, and retention/folder/delete workflows can keep hitting lock conflicts until an admin identifies the specific document and manually unlocks it. This is not byte corruption, but it creates a persistent stuck collaboration state with weak production recovery ergonomics.

Recommendation: add an explicit stale-lock workflow: lock age visibility for admins, configurable stale thresholds, notification/escalation, and an audited force-release action that does not depend on the original owner returning. If locks are intended to be indefinite, document the operational runbook and surface all active locks in an admin view. Add tests for disabled/missing lock owners and for stale-lock discovery/release.

### Edit Workflow - Lock acquisition can commit after stale write/access checks

The explicit lock API checks editability before opening the transaction, then inserts an active lock by document id without reloading the document row or current folder access inside the transaction (`vault/server/src/documents.rs:239-283`). Checkout has the same shape after the object read: `record_checkout_event_and_lock` reloads editability before the transaction, then only checks the active lock row before inserting or accepting the lock (`vault/server/src/http.rs:1484-1494`, `vault/server/src/documents.rs:590-634`). Folder permission changes, archive/restore, move/rename, and admin user disablement are separate transactions.

Impact: a user can pass the pre-check while they still have write access to an active Vault document, then a concurrent permission change, archive, move, or user-disable can commit before the lock transaction inserts its row. The lock can remain active on a document the caller no longer has write access to or can no longer even see. Because unlock also starts with `document_for_write` before it releases the row (`vault/server/src/documents.rs:285-322`), the stale lock owner may be unable to clear their own lock after access is revoked. That can block other writers and can keep retention pending until an admin intervenes.

Recommendation: make lock acquisition a guarded write transaction: reload the document, folder root/archive state, current access, user active status, and active-lock row inside `BEGIN IMMEDIATE`, then insert only if all predicates still hold. For checkout, acquire the lock before object reads as noted above, and release or mark it if the later storage response fails. Add race tests for lock-vs-permission-removal, lock-vs-archive, checkout-vs-archive, and unlock-after-access-revocation.

### SQLite Contention - Normal reads execute write statements during auth and folder resolution

Most authenticated routes call `current_user`, including bootstrap/settings reads, SSE stream connection, folder listing, and downloads (`vault/server/src/http.rs:745-775`, `vault/server/src/http.rs:1022-1065`, `vault/server/src/http.rs:1098-1108`, `vault/server/src/http.rs:1471-1481`). In OIDC mode, `current_user` calls `session_identity` (`vault/server/src/http.rs:2644-2657`). `session_identity` verifies the signed cookie, fetches the user, rejects disabled users, and then performs `UPDATE vault_users SET last_seen_at = ? WHERE id = ?` on every request before returning context (`vault/server/src/auth.rs:404-430`). In header mode, the same `current_user` path calls `header_identity`, which upserts the user and syncs memberships/groups from headers on every request (`vault/server/src/auth.rs:339-380`, `vault/server/src/auth.rs:524-599`, `vault/server/src/auth.rs:618-657`).

Folder path resolution adds another write-shaped read path. `get_root_folder` always executes `INSERT OR IGNORE INTO folders ...` before fetching the root row (`vault/server/src/folders.rs:234-254`). `get_folder_by_path` calls it for ordinary path lookup (`vault/server/src/folders.rs:267-286`), and read payload builders call those helpers for bootstrap, contents, sidebar, folder properties, and access checks (`vault/server/src/views.rs:288-360`, `vault/server/src/views.rs:386-397`, `vault/server/src/views.rs:568-578`). The roots are seeded at startup, so these steady-state read requests should not need to write.

Impact: read-heavy traffic is turned into write traffic against the single SQLite writer. OIDC deployments can see ordinary GETs, SSE reconnects, folder listings, and download starts contend with uploads, admin changes, sweeps, export progress writes, and each other. Header-auth deployments are heavier because group sync runs in an immediate transaction every request. The 30s busy timeout and 10-connection pool reduce fast failures (`vault/server/src/db/mod.rs:22-32`), but they also allow request latency to pile up, and these hot-path writes have no targeted retry or best-effort fallback.

Recommendation: take `last_seen_at`, header identity sync, and root-folder seeding off the per-request critical path. Throttle user timestamp updates, update asynchronously/best-effort, or only persist when the prior timestamp is older than a small interval. Fetch seeded root folders with `SELECT` after startup instead of `INSERT OR IGNORE` on normal reads. If header auth remains request-based, avoid rewriting memberships unless the incoming claim set changed. Add a concurrency test with many authenticated GET/download/SSE starts while a write transaction is active.

### Availability - Folder/search views scan and materialize unbounded document state

The main folder contents endpoint has no pagination or server-side result limit (`vault/server/src/http.rs:1056-1065`). `build_contents_payload` loads all folders, all active locks, and all visible document rows before filtering to the requested folder/search in Rust (`vault/server/src/views.rs:386-423`). The document list helper runs a full `SELECT ... FROM documents` with joins and `fetch_all` but no `WHERE`, `LIMIT`, or `OFFSET` (`vault/server/src/views.rs:1238-1277`), and the lock helper also loads every active lock with `fetch_all` (`vault/server/src/views.rs:1484-1503`). Access checks then add per-folder/per-document work: `visible_document_rows` loops over every document and calls `document_access_level`, which can call `folder_access_level`; non-admin folder access walks ancestors and queries permissions for each ancestor (`vault/server/src/views.rs:638-652`, `vault/server/src/documents.rs:1044-1056`, `vault/server/src/folders.rs:758-795`).

Other UI payloads reuse the same unbounded helpers. Sidebar construction loads all folders and checks access for each (`vault/server/src/views.rs:288-331`), my-edits and favorites load all folders/documents/locks before filtering (`vault/server/src/views.rs:437-471`, `vault/server/src/views.rs:784-854`), and folder properties can load all visible documents just to compute counts (`vault/server/src/views.rs:568-635`). The browser request shape has folder/search parameters but no page/limit controls (`vault/client/src/lib/useVaultResources.js:207-255`).

Impact: as a production vault grows, ordinary browsing, search, bootstrap refreshes, and SSE-triggered refreshes can become full-database scans with large allocations and large JSON responses. A broad recursive search or repeated tab refresh can monopolize the SQLite pool/CPU and make unrelated uploads, downloads, auth requests, and sweeps look hung. This is not metadata corruption, but it is a realistic availability ceiling for serious production use.

Recommendation: add pagination/result caps and push folder/search/access filtering into SQL where practical. Cache or batch permission evaluation instead of doing ancestor permission queries per row, and make sidebar/count/history payloads bounded or lazy-loaded. Add load tests with large folder/document/lock counts that assert bounded query counts, response size, and latency.

### Availability - Favorite preferences have no item cap and are fully resolved on every preferences read

The preferences validator is strict about allowed keys and types, but it does not cap `favoriteItems`. `clean_favorite_items_strict` accepts every valid folder/document id, de-duplicates with a growing `Vec`, and returns the entire array for persistence (`vault/server/src/preferences.rs:227-250`). `update_preferences_for_user` writes that normalized JSON directly into `vault_users.preferences` (`vault/server/src/preferences.rs:45-76`). The normal client path also has no count limit when it normalizes or sends favorites (`vault/client/src/lib/favoriteItems.js:32-44`, `vault/client/src/lib/theme.js:207-221`).

Every preferences response then resolves that persisted list instead of returning ids. `api_preferences` and the update response both call `build_preferences_payload` (`vault/server/src/http.rs:1022-1045`), which loads all folders, all visible documents, folder stats, and active locks before iterating every stored favorite (`vault/server/src/views.rs:784-854`). Per favorite, folder entries call `folder_access_level`, and document entries linearly search `visible_docs` before calling `document_access_level` (`vault/server/src/views.rs:856-918`). This means one large saved preference can repeatedly force large allocations, repeated permission queries, and a large JSON response on bootstrap, explicit preference refresh, and every preference update.

Impact: an authenticated user can persist a self-reinforcing slow state for their account without uploading files or needing admin privileges. It is not database corruption, but it is a durable faulty state: the account's normal UI bootstrap/preferences refresh can become slow, memory-heavy, and SQLite-writer-contending enough to look hung. In a shared single-container SQLite deployment, repeated reconnects or SSE-triggered preference refreshes from that user can consume capacity needed by unrelated uploads, downloads, and admin operations.

Recommendation: add an application-level maximum favorite count, reject or truncate oversized patches with a clear error, and use a `HashSet` for duplicate detection. Resolve favorite ids with bounded SQL queries rather than loading the whole vault and looping per item, and return only a capped set in bootstrap/preferences payloads. Add tests for oversized favorite patches and for preferences reads with many favorites proving bounded query count, response size, and latency.

### Fault Isolation - One inconsistent current-version pointer can break broad user views

The schema does not make `documents.current_version_id` a foreign key to `document_versions.id`, and the view layer treats mismatches as a fatal view error. `document_view_rows` loads every document with a left join to only `d.current_version_id`; `visible_document_rows` then loops through all rows visible to the user and returns `InconsistentDocumentVersion` if any visible document has a non-empty current pointer, or any version rows, but no joined latest version (`vault/server/src/db/schema.rs:119-177`, `vault/server/src/views.rs:638-648`, `vault/server/src/views.rs:1210-1277`). Broad payloads call that helper for folder contents, share-folder resolution, folder properties/counts, preferences, and initial app-shell state (`vault/server/src/views.rs:367-383`, `vault/server/src/views.rs:386-423`, `vault/server/src/views.rs:543-590`, `vault/server/src/views.rs:784-854`). Route tests assert that a single document with `current_version_id = 'missing-version'`, a stale pointer plus `version_count = 0`, or a null pointer while versions exist makes `/api/folders/contents` return a 500 `"Current document version metadata is inconsistent"` (`vault/server/tests/routes_folders.rs:680-755`).

Impact: any bug, interrupted manual migration, failed repair, or external DB edit that leaves one visible document's version pointer inconsistent can take down broad browsing/bootstrap/preferences/share-folder payloads for users who can see that row. The failure is deliberate and safer than silently serving the wrong bytes, but it has poor fault isolation: healthy folders and documents become unavailable until the bad row is repaired, and the current Docker healthcheck/readiness path will not catch it before users hit the UI.

Recommendation: keep the hard consistency signal, but isolate it. Add a startup/admin/readiness consistency check that reports exact bad document ids, and make broad list payloads either omit poisoned rows with an explicit per-row error marker or return a structured partial-failure response instead of failing the entire payload. Add a repair/admin workflow to repoint `current_version_id` to the newest valid version or quarantine the document. Keep direct document detail/download strict for the affected row.

### Transfer State - Abort/cancel/expiry can be overwritten by stale completion or sweep transitions

Several transfer state transitions read status, do work, and then update by `id` without rechecking the expected current status. Upload completion moves a session to `completing` with a guarded update (`WHERE id = ? AND status = 'active'`), but the final completion update sets `status = 'complete'` with only `WHERE id = ?` (`vault/server/src/uploads.rs:2005-2034`, `vault/server/src/uploads.rs:933-956`). `abort_upload_session` similarly fetches the session once and, if it was not complete at that time, sets `status = 'aborted'` with only `WHERE id = ?` (`vault/server/src/uploads.rs:831-859`). The transfer sweeper selects expired upload/export rows before its transaction and later updates by `id` without status guards (`vault/server/src/transfers.rs:286-324`, `vault/server/src/transfers.rs:408-473`).

Exports have a related cancellation/finalization race. `cancel_export_job` fetches a row and updates `status = 'cancelled'` by `id` only (`vault/server/src/exports.rs:596-620`). `complete_export_job` checks cancellation, then sets `status = 'finalizing'` by `id` only, checks again, and finally completes only if status is still `finalizing` (`vault/server/src/exports.rs:828-836`, `vault/server/src/exports.rs:1382-1451`). Existing tests cover cleanup when cancellation lands during artifact promotion (`vault/server/tests/routes_exports.rs:1723-1863`), but they do not cover stale status overwrites by abort/sweep/finalizing updates.

There is also a non-race completion overwrite. After `persist_export_artifact` sets `export_jobs.status = 'complete'`, `complete_export_job` records per-document export/download history; if that history insert fails, the function returns an error (`vault/server/src/exports.rs:796-842`, `vault/server/src/exports.rs:1433-1452`, `vault/server/src/exports.rs:1541-1562`). The outer spawned worker catches that error and calls `mark_export_failed`, whose guard is only `status != 'cancelled'`, so it can rewrite an already completed export to `failed` (`vault/server/src/exports.rs:622-655`, `vault/server/src/exports.rs:1604-1623`).

Impact: these races should not corrupt document version metadata by themselves because canonical upload document writes are transactional and export artifacts are derived. They can still produce faulty user-visible state: an abort can be lost to a concurrent upload completion, a just-completed upload session can be marked expired by a stale sweep, a completed export can be made inaccessible by a stale expiry/cancel update, an export cancellation can be overwritten by finalizing, or a successfully persisted export can appear failed because audit/history logging failed afterward. These are exactly the kind of hard-to-reproduce production states that make transfer support look wedged or inconsistent.

Recommendation: make every terminal transition conditional on the expected prior state (`WHERE id = ? AND status IN (...)`), check `rows_affected`, and map lost races to a stable response by reloading current state. For sweeps, either move the selection into the same immediate transaction or include the selected status in guarded updates. Never let post-completion history/event logging turn a completed export into failed; either record those events before the terminal completion update, make them best-effort after completion, or guard `mark_export_failed` to only affect queued/running/finalizing jobs. Add race tests for complete-vs-abort, complete-vs-expire, cancel-vs-finalizing, sweep-vs-complete, and completed-export history insertion failure.

### Transfer State - Upload completion can get stuck after publishing `completing`

The upload part path writes the request body to a temporary file, atomically promotes it to `000000NN.part`, and only then writes `000000NN.json` metadata when the request supplied `X-Upload-Sha256` (`vault/server/src/uploads.rs:620-668`). If `write_part_metadata` fails after promotion, the part file remains but its checksum metadata is absent. A retry with the same checksum hits the existing-part branch, reads fallback metadata from file size only (`sha256: None`), and `part_checksum_headers_match` rejects `(None, Some(incoming))` as a conflict (`vault/server/src/uploads.rs:652-664`, `vault/server/src/uploads.rs:1477-1494`, `vault/server/src/uploads.rs:1753-1789`). Tests intentionally cover the same state shape for a no-checksum upload followed by a checksum duplicate, which returns `409 CONFLICT` (`vault/server/tests/routes_uploads.rs:784-865`), but I did not find coverage for the failure window after checksum promotion and before metadata persistence.

There is a broader completion state gap around the same pre-commit phase. `complete_upload_session_impl` first changes an active session to `completing`, then calls `completed_parts` / `UploadHashCoordinator::completed_parts` with `?` before entering the `complete_upload_session_inner` path whose error branch calls `mark_upload_failed` (`vault/server/src/uploads.rs:781-825`, `vault/server/src/uploads.rs:1275-1285`, `vault/server/src/uploads.rs:2005-2034`, `vault/server/src/uploads.rs:2100-2121`). Missing parts, scratch I/O errors, hash-coordinator missing digest, or a full-file checksum mismatch can therefore return after the row is already `completing` without marking it `failed` or clearing the coordinator state. A retry then fails `ensure_active_session` because `completing` is no longer an active upload, so the caller cannot simply retry the completion request from the original session state (`vault/server/src/uploads.rs:797-813`, `vault/server/src/uploads.rs:2073-2088`).

Impact: an API client that uses the stronger checksum hooks can receive an error for a part that was actually promoted, then be unable to retry that part with the same checksum. Separately, a bad checksum, missing scratch file, or transient filesystem read failure during completion can leave the upload session stuck in `completing` until the user aborts, the session expires, startup recovery intervenes, or an operator reconstructs/removes scratch metadata. This is not committed document corruption, but it is a durable faulty transfer state; for check-ins it can also keep the surrounding edit workflow unresolved because the check-in lock is only released by successful completion.

Recommendation: make the part file and checksum metadata publish atomically from the retry contract's perspective. Options include writing metadata before the final promote and treating a same-size existing part with no metadata as verifiable by hashing it against the incoming checksum, or storing checksum metadata in a sidecar that is created with the part under a single temporary directory rename. Also wrap the post-`mark_upload_completing` verification phase so any pre-store failure either reverts to `active` when safe or marks the session `failed` with a clear error and clears process-local hash state. Add regression tests that inject/fake metadata-write failure after promotion, retry with the same checksum, and force completion-time missing-part/checksum failures after the row is already `completing`.

### Transfer Recovery - Browser uploads interrupted during completion are failed instead of recovered

Upload completion changes the session from `active` to `completing` before it hashes parts, writes the object, and commits the canonical document/version update (`vault/server/src/uploads.rs:781-815`, `vault/server/src/uploads.rs:2005-2034`). That gives startup recovery a real crash/restart window where all uploaded part files can exist, but the database still has only an in-progress upload session.

Normal browser uploads do not write per-part JSON metadata because `write_part_metadata` is only called when a part supplied `sha256` (`vault/server/src/uploads.rs:645-668`), and the browser client does not send `X-Upload-Sha256` (`vault/client/src/lib/transferClient.js:227-288`). The normal completion path can still reconstruct unchecked parts from file size when `.json` is missing (`vault/server/src/uploads.rs:1739-1778`). Startup recovery does not use that same parser: `upload_has_recoverable_parts` requires every `000000NN.json` file and returns unrecoverable if the metadata is missing (`vault/server/src/transfers.rs:362-405`). The recovery test fixture for a "recoverable" upload writes both `.part` and `.json`, so it does not cover the first-party browser path (`vault/server/tests/transfers.rs:209-228`, `vault/server/tests/transfers.rs:310-376`).

There is a second client-side recovery gap after the server has actually committed. Completed upload sessions are idempotently representable: `UploadSessionPayload` includes `result`, `upload_session_payload` returns it for completed rows, and `complete_upload_session_impl` returns `completed_result` when called again for a `complete` session (`vault/server/src/uploads.rs:102-116`, `vault/server/src/uploads.rs:781-796`, `vault/server/src/uploads.rs:1640-1697`, `vault/server/src/uploads.rs:2293-2305`). The browser stores the session id in `localStorage`, but on the next attempt it only reuses a stored session when `session.status === "active"`; any `complete` session fetched through `existingUploadSession` is ignored and replaced with a new upload session (`vault/client/src/lib/transferClient.js:400-421`). If the completion POST commits the document/version and the response is lost, or the tab reloads before `forgetUploadSession`, the browser cannot surface the already-completed result and instead may reupload against a path or check-in state that already changed (`vault/client/src/lib/transferClient.js:520-540`).

Impact: a container restart, crash, OOM kill, or deploy that lands after `mark_upload_completing` but before the final upload commit can turn a fully transferred browser upload into `failed` on startup and delete the scratch directory. For create uploads this loses the user's transferred data and forces a full reupload; for check-ins it can also leave the existing document lock and edit workflow unresolved until the user retries or an admin intervenes.

Even when the server-side commit succeeds, the browser-side lost-response path turns a successful transfer into a confusing retry/failure loop: the canonical object/version exists, but the client cannot reconcile that success from the saved session id. In create mode the retry can collide with the document it already created; in check-in mode it can collide with the lock/version state that the successful completion already advanced.

Recommendation: make recovery use the same `read_part_metadata` fallback that normal completion uses, or write metadata for every part regardless of whether the client supplied a checksum. In the browser, treat a stored completed session with a `result` as success, clear the saved session, and refresh the relevant view instead of creating a replacement session. Add a restart/recovery test for a `completing` browser-style upload with only `*.part` files and no `*.json` files, and add a client/API test for "completion response lost but `GET /api/uploads/:id` returns `complete` with result".

### Startup Recovery - Interrupted transfer recovery is unbounded and can hold a write transaction during filesystem scans

Startup runs transfer sweep and interrupted-transfer recovery before binding the listener (`vault/server/src/main.rs:31-45`). The sweep path is bounded to 250 expired upload/export rows per run, but interrupted recovery fetches every non-expired `upload_sessions.status = 'completing'` row and every `export_jobs.status IN ('running', 'finalizing')` row without a limit (`vault/server/src/transfers.rs:156-165`, `vault/server/src/transfers.rs:326-359`). It then opens one SQLite transaction for the whole recovery batch (`vault/server/src/transfers.rs:170-237`).

For uploads, that transaction loop calls `upload_has_recoverable_parts` before each row update; the helper performs filesystem reads and metadata checks for every expected part (`vault/server/src/transfers.rs:170-205`, `vault/server/src/transfers.rs:362-405`). The first upload's scan happens before SQLite has necessarily promoted the deferred transaction to a writer, but after the first `UPDATE`, later per-part filesystem scans happen while the recovery transaction and connection are still open. On a crash with many large completing sessions, a slow `/data` volume, or a large part-count backlog, startup can spend a long time before serving and can hold the write transaction while doing disk work.

The same helper treats malformed JSON or non-`NotFound` filesystem errors as fatal recovery errors rather than marking that upload failed. It reads each `*.json` metadata file and deserializes it with `serde_json::from_slice`, propagating parse/read errors through `?` (`vault/server/src/transfers.rs:362-405`). Those metadata files are written by `fs::write` plus rename without durability checks (`vault/server/src/uploads.rs:1792-1805`). Since `main` awaits recovery with `?` before the listener is bound, one bad metadata file for a non-expired `completing` upload can make every container restart fail until an operator manually repairs the row or scratch directory (`vault/server/src/main.rs:36-44`).

Impact: after a production restart, the service can appear hung or fail health checks while it synchronously scans an unbounded interrupted-transfer set. In a single-container deployment no user traffic is served yet, but Docker/nginx will see a prolonged unhealthy startup; in an accidental multi-replica deployment this startup writer can block the live instance. A single corrupt transfer metadata file can also turn an otherwise discardable interrupted upload into a repeatable boot failure. This is a poor recovery failure mode for the same transfer-backlog and scratch-space exhaustion scenarios described elsewhere.

Recommendation: bound interrupted recovery per startup pass, do filesystem validation outside a SQLite write transaction, then reopen a short transaction to conditionally update each row by expected status. Treat malformed/unreadable per-upload scratch metadata as a per-row non-recoverable upload failure, not a process startup failure, and keep operator-visible diagnostics. Continue recovery in the background after readiness, with operator-visible progress. Add tests for corrupt `*.json` metadata and a benchmark for many completing sessions with many parts to prove startup recovery is bounded and does not hold the write transaction across filesystem scans.

### Operations - No graceful shutdown or transfer drain for Docker restarts/deploys

Startup does recovery work, but shutdown is not coordinated. `main` spawns the TTL sweeper, binds the listener, and awaits `axum::serve(listener, app)` directly; I did not find `with_graceful_shutdown`, SIGTERM/SIGINT handling, or a server-drain path for in-flight uploads, downloads, and export workers (`vault/server/src/main.rs:31-51`, `vault/server/src/main.rs:55-74`). The Compose service uses `restart: unless-stopped` but does not set an application-specific drain/stop procedure (`docker-compose.yml:1-8`). Export jobs and upload hash preverification are spawned process-local tasks (`vault/server/src/exports.rs:622-655`, `vault/server/src/uploads.rs:309-320`).

Impact: routine production deploys, Docker restarts, host reboots, and `docker compose stop` can look like abrupt crashes to active transfers. SQLite should protect committed metadata, but active uploads can fall into the recovery gaps above, export artifacts can be interrupted and later marked failed, remote-storage staging can leave `/tmp` files, and users get avoidable failed or duplicated transfer work during normal operations.

Recommendation: add graceful shutdown that stops accepting new requests, signals SSE clients, cancels or drains export workers within a bounded deadline, prevents new upload completions from starting, and lets in-flight upload part writes finish or be marked recoverable before process exit. Document Docker `stop_grace_period` and deploy procedures, and add integration tests that send shutdown during upload completion/export finalization without creating unrecoverable transfer state.

### Data Integrity - Browser resumable uploads do not use the server checksum hooks

The upload API supports optional per-part and full-object SHA-256 checks, but the browser client does not send them in its normal resumable flow. `uploadPartRequest` sends content type, offset, size, and optionally the upload token, but no `X-Upload-Sha256` header (`vault/client/src/lib/transferClient.js:227-288`). Completion supports a `sha256` body field, yet `uploadFileResumable` always calls `completeUploadSession(session, null, signal)` (`vault/client/src/lib/transferClient.js:341-351`, `vault/client/src/lib/transferClient.js:520-523`). The local resume key is based on mode, document/folder, name, size, mtime, note, and rename flag, not file content (`vault/client/src/lib/transferClient.js:144-154`).

On the server, part hashing only happens when the request supplied an expected SHA-256 (`vault/server/src/uploads.rs:1513-1554`), and final upload digest verification only rejects when completion supplied `expected_sha256` (`vault/server/src/uploads.rs:1275-1285`). For idempotent retries, an existing part without metadata is accepted when its size matches and neither side has a checksum (`vault/server/src/uploads.rs:1477-1494`, `vault/server/src/uploads.rs:1753-1778`). Tests reflect that split: no-checksum part uploads are accepted and create no `.json` metadata, while the conflict-rejection test uses a helper that always sends `x-upload-sha256` (`vault/server/tests/routes_uploads.rs:784-837`, `vault/server/tests/routes_uploads.rs:412-429`, `vault/server/tests/routes_uploads.rs:983-1030`).

Impact: this is not a normal TLS/TCP corruption path, because the application stores the bytes it receives and later computes a content-addressed digest over completed parts. The production risk is stale or conflicting resumable state: if a retry/resume path, local scratch corruption, or same-size duplicate part conflict leaves different bytes in an existing unchecked part slot, the browser flow can accept the stale bytes and complete a document that differs from the currently selected file. Because the first-party client never sends the full expected digest, completion cannot catch that mismatch.

Recommendation: compute and send per-part hashes from the browser, or at least compute and send the full-file SHA-256 before completion and reject mismatches. If browser-side hashing cost is a concern, make no-checksum resumable uploads non-idempotent for conflicting existing parts, add a server-side stored digest/etag for unchecked parts after first write, and cover same-size no-checksum duplicate conflicts in tests.

### Data Lifecycle - Create-upload completion can resurrect an archived folder path

Create upload sessions store a normalized `folder_path` string, not the target folder id or a folder generation/version (`vault/server/src/uploads.rs:419-475`). Session creation does create or validate the folder path and checks write access at that moment (`vault/server/src/uploads.rs:493-509`). Completion later rechecks write access for the stored path before opening the canonical write transaction, but `require_write_for_folder_path` authorizes the nearest existing ancestor if the full path no longer exists (`vault/server/src/uploads.rs:897-914`, `vault/server/src/folders.rs:799-827`). The create completion transaction then calls `get_or_create_folder_path_in_tx` for the stored path and inserts the document there (`vault/server/src/uploads.rs:988-1037`).

That makes a stale upload session able to recreate a path that a normal folder archive just removed. `archive_folder` archives the documents it precomputed and then deletes the source folder ids (`vault/server/src/documents.rs:745-815`, `vault/server/src/documents.rs:1466-1479`). If an upload session targeting that folder was already active and completes after the archive transaction commits, the missing Vault folder path can be recreated under any still-writable ancestor, and the new document lands outside Archive even though the folder was just archived.

Impact: a user can see a folder archive succeed, then a late resumable upload can bring the same Vault path back with new active content. That does not corrupt bytes, but it breaks the lifecycle expectation that archiving a folder drains that subtree and removes it from active Vault. It can also bypass operator cleanup intent during incident response or retention work when users have long-lived upload sessions.

Recommendation: store the intended target folder id and expected path/generation on create-upload sessions, then re-fetch and verify that exact folder still exists, is still in Vault, and is still writable inside the completion transaction. If the folder was archived, moved, deleted, or replaced, fail the session with a stale-target conflict instead of recreating the path. Add a race test for create-upload-session-vs-folder-archive and a stale missing-folder completion test.

### Data Integrity - Check-in completion can apply stale document folder/name state after a concurrent move

Check-in completion preloads the editable document, write authorization, and result path before opening the SQLite transaction (`vault/server/src/uploads.rs:897-914`). Inside the transaction, `complete_checkin_upload_in_tx` uses that preloaded `DocumentRecord` for duplicate-name checks, version creation, lock release, and result path generation (`vault/server/src/uploads.rs:1040-1097`). The final release helper updates only `documents.name`, `latest_modified_at`, and `latest_modified_by`, then reapplies TTL with the preloaded `document.folder_id`; it does not re-fetch the current document row or verify that the folder/name/access state still match the snapshot (`vault/server/src/uploads.rs:1134-1157`). Serial tests cover an archive before completion starts (`vault/server/tests/routes_uploads.rs:1804-1886`), but I did not find a race test for a move/rename or permission revocation after the preload and before the check-in transaction.

The lock does not prevent that race for the lock owner or an admin. Document move/rename checks an active lock but allows the lock owner/admin to proceed (`vault/server/src/documents.rs:888-891`), then moves the document and reapplies TTL to the destination folder (`vault/server/src/documents.rs:915-945`). Therefore a same-user/admin move can commit between check-in's pre-transaction fetch and its write transaction. The later check-in can then create the new version against the current document id while applying the old folder's TTL policy and, for non-rename check-ins, writing the old name back over a concurrent rename. The TTL helper writes directly to the document id using the caller-supplied folder policy (`vault/server/src/folders.rs:691-722`).

Impact: a normal lock-owner workflow can leave the document in the new folder but with retention metadata from the old folder, or can lose a concurrent rename. The worst case is moving a locked document out of a short delete-TTL folder to a safe folder while check-in completion reattaches the old delete policy; the document can later be deleted by retention even though the visible folder no longer has that policy. This is a faulty persistent state rather than immediate SQLite corruption.

Recommendation: perform the check-in document reload, archive-state check, write-access check, lock-owner check, duplicate-name check, version creation, document update, TTL application, and upload-session completion in one `BEGIN IMMEDIATE` transaction. Use the reloaded current `folder_id` and `name` for TTL/result paths, or reject with a stale-state conflict if the row changed after session creation. Add race tests for check-in-vs-move, check-in-vs-rename, check-in-vs-permission-revocation, and check-in-vs-move-out-of-delete-TTL.

### Authorization - Queued export/download jobs use a serialized privilege snapshot instead of live user state

Export job creation serializes the entire `UserContext` into `export_jobs.user_context` (`vault/server/src/exports.rs:511-570`, `vault/server/src/exports.rs:1709-1719`). The worker later claims the job, deserializes that JSON back into a `UserContext`, and resolves downloads with it (`vault/server/src/exports.rs:796-807`, `vault/server/src/exports.rs:974-1008`). That context includes `groups` and `is_admin` (`vault/server/src/auth.rs:298-308`). Access checks then use those stale fields directly: folder access grants admin override and compares `user.groups` to live folder permissions (`vault/server/src/folders.rs:758-795`), while document downloads call `document_for_read` and `document_access_level` through the same user context (`vault/server/src/documents.rs:437-467`, `vault/server/src/documents.rs:1088-1104`).

The queued work also stores selected document/folder ids as bare JSON, not as constrained rows or immutable version ids (`vault/server/src/exports.rs:194-205`, `vault/server/src/exports.rs:538-570`). The worker replays those ids later (`vault/server/src/exports.rs:974-1008`). `folders.id` and `documents.id` are plain `INTEGER PRIMARY KEY` rowids, not `AUTOINCREMENT` or UUIDs (`vault/server/src/db/schema.rs:7-23`, `vault/server/src/db/schema.rs:119-140`), and `export_jobs.request_payload` has no foreign key to the originally selected rows (`vault/server/src/db/schema.rs:246-258`). A queued job can therefore outlive a deleted target and later resolve the same numeric id if SQLite reuses it for a new folder/document.

Impact: if an admin disables a user, removes the user from a group, renames/removes an admin group, or revokes stored admin status after a queued export/download job is created but before the worker resolves it, the worker can still build the archive with the old privileges. Normal OIDC requests re-check `vault_users.is_active` through `session_identity` (`vault/server/src/auth.rs:404-430`), but queued workers do not reload the user or group memberships before doing data access. If a selected row is deleted and its numeric id is reused before the worker runs, the job can also export a different current document/folder than the user selected, under that same stale privilege snapshot.

Recommendation: persist only a stable requester identity on the job, then reload an active current user context inside the worker before resolving downloads. Store immutable selected targets: version ids for document downloads, stable UUID-style row identifiers, or normalized paths plus expected row/version checks that fail if the target changed. Treat disabled users, lost access, deleted targets, and id-reuse mismatches as cancellation/failure. Add tests that revoke a queued user's group/admin status, disable the user, and delete/recreate a selected max-rowid document/folder before a delayed worker runs.

### Authorization - Archive access snapshots can be stale under concurrent permission changes

Archived document access is intentionally evaluated from two sources: current access to the Archive root and the stored `documents.archived_access` JSON snapshot of source-folder group access (`vault/server/src/documents.rs:1020-1042`). However, single-document archive computes `archive_access_snapshot` before opening the mutation transaction, and folder archive computes each descendant document's snapshot before the transaction that moves documents into Archive and deletes the source folders (`vault/server/src/documents.rs:637-673`, `vault/server/src/documents.rs:745-815`). Retention archive does the same: it selects expired rows, builds path/access plans, and only then opens the write transaction (`vault/server/src/documents.rs:352-430`). Folder permission replacement is a separate transaction that deletes and reinserts permissions for the folder (`vault/server/src/folders.rs:445-510`).

The snapshot is also not relationally tied to the group rows it names. `archive_access_snapshot` serializes a `HashMap` keyed by `vault_groups.id`, and `archived_access_level` later walks current groups, matches them to the caller's current group names, and looks up that current group id in the old JSON snapshot (`vault/server/src/documents.rs:991-1042`). The JSON column has no foreign key to `vault_groups`, and group ids are ordinary SQLite rowids (`vault/server/src/db/schema.rs:54-60`, `vault/server/src/db/schema.rs:119-140`). Admin deletion only checks live `folder_permissions` before deleting a group (`vault/server/src/admin.rs:293-318`); it does not know whether old `documents.archived_access` JSON still references that id. If a deleted group id is later reused by a different group name, Archive access can be evaluated against stale source permissions that belonged to the deleted group rather than the current principal. The automatic root permission creation for every synced group makes that more likely to matter because current Archive-root access is easy to obtain (`vault/server/src/auth.rs:524-630`).

Impact: if an admin changes source-folder permissions while an archive operation is between snapshot calculation and commit, the archived row can persist an ACL snapshot that never matched the committed moment of archive. A just-revoked group can retain archive access if it also has Archive-root access; a just-granted group can be omitted. Later group deletion/recreation can also make an archived row's JSON snapshot apply to the wrong current group id. The document bytes and version metadata are not corrupted, but the archived authorization state can be wrong indefinitely because reads consult the stored snapshot rather than live source-folder permissions or stable external group identities.

Recommendation: compute archive ACL snapshots inside the same transaction that archives the document and re-check source folder write/access state there. Store stable canonical group keys in `archived_access`, or use a normalized snapshot table with foreign keys/tombstones instead of row-id JSON. For folder archive and retention sweeps, build candidate lists outside the transaction if needed, then reload each document/folder permission snapshot inside a short guarded transaction before writing `archived_access`. Add tests where folder permissions change while archive is paused before commit and where an archived source group is deleted/recreated before a user resolves an Archive document.

### Authorization / Data Lifecycle - Folder path updates can apply to a stale folder after concurrent moves

The folder settings APIs identify the target by path, not by a stable id/version. The appearance, permissions, and retention endpoints accept `payload.path` and pass it into separate update helpers (`vault/server/src/http.rs:1122-1185`). `update_folder_properties` resolves the path and access level before its transaction, then updates by the pre-resolved `folder.id` (`vault/server/src/folders.rs:405-442`). `update_folder_permissions` resolves the path to a `FolderRecord` before opening its write transaction, validates group ids outside that transaction, and then replaces rows for the pre-resolved `folder.id` (`vault/server/src/folders.rs:445-510`). `update_folder_retention` resolves the path, builds the subtree and access checks outside the transaction, then writes retention settings and reapplies TTLs for the precomputed subtree ids (`vault/server/src/folders.rs:513-572`, `vault/server/src/folders.rs:1241-1285`). Folder move/rename uses independent transactions that can change `folders.parent_id` and `name` for the same id (`vault/server/src/folders.rs:575-683`).

Impact: if a folder is moved or renamed between the path lookup and the settings transaction, a request intended for `Project/Secret` can commit ACL, appearance, or retention rows to the folder that used to live at that path, not necessarily the folder currently visible at that path. A concurrent replacement folder at the old path would keep its previous settings, while the moved folder receives the new grants/revocations or TTL policy. For retention, this can place an archive/delete TTL on the wrong subtree and later drive unintended archive or deletion through the retention sweep. The response rebuilds properties for the original path rather than proving the mutated folder still has that path.

Recommendation: make folder settings updates target a stable folder id plus a last-seen path/version, or re-resolve and verify the path inside a short `BEGIN IMMEDIATE` transaction immediately before mutating settings. Retention updates should also compute the subtree and apply TTLs from the transaction's current folder graph. If the folder id/path changed, return a conflict and force the UI to reload. Add race tests for permission and retention updates versus folder move/rename and replacement of the original path.

### Authorization - OIDC group and profile changes are only synced on login/callback

The OIDC callback exchanges the code, merges ID token/userinfo claims, extracts the configured groups claim, and calls `oidc_identity`, which upserts the user and syncs group memberships (`vault/server/src/oidc.rs:95-138`, `vault/server/src/auth.rs:432-450`, `vault/server/src/auth.rs:524-595`). That sync removes memberships for groups no longer present in the current login claim set (`vault/server/src/auth.rs:618-657`).

Normal OIDC requests do not refresh claims from the IdP. `current_user` in OIDC mode only calls `session_identity` (`vault/server/src/http.rs:2644-2657`). `session_identity` verifies the signed local session cookie, fetches `vault_users` by `uid`, rejects locally inactive users, updates `last_seen_at`, and builds `UserContext` from stored group memberships (`vault/server/src/auth.rs:404-430`, `vault/server/src/auth.rs:707-739`). The local session cookie is issued with `VAULT_SESSION_MAX_AGE_SECONDS`, defaulting to 604800 seconds/seven days (`vault/server/src/auth.rs:205-208`, `vault/server/src/http.rs:512-528`).

Impact: if an IdP administrator removes a user from a group, disables the upstream account, changes email/name, or removes admin-group membership, an already-issued Vault session keeps using the stored local identity and memberships until the user logs in again, the cookie expires, or a Vault admin updates/disables the local user. Because folder permissions are evaluated against stored Vault group memberships, OIDC-side access revocation is not live for active sessions. This can leave stale write/read/admin access for days in the default configuration.

Recommendation: document OIDC claim revocation latency clearly and reduce the default session max age for high-trust deployments, or add a periodic/TTL-based OIDC claim refresh/introspection check before authorizing requests. At minimum, provide an admin-visible "last synced from IdP" timestamp and a way to force logout/disable all sessions for a user after IdP revocation.

### Concurrency - Most write transactions are deferred; only identity sync explicitly queues writers

Identity upsert uses `BEGIN IMMEDIATE` and retry logic to avoid login-burst write promotion failures (`vault/server/src/auth.rs:524-599`). Most other mutating flows use `pool.begin()` with read/check/write sequences: lock/delete/retention flows (`vault/server/src/documents.rs:340`, `vault/server/src/documents.rs:409`), folder create/move/retention (`vault/server/src/folders.rs:358`, `vault/server/src/folders.rs:646`), upload finalization (`vault/server/src/uploads.rs:914`), and transfer sweep/recovery (`vault/server/src/transfers.rs:103`, `vault/server/src/transfers.rs:170`).

Impact: SQLite constraints protect canonical consistency in many cases, and failed transactions roll back, so this is unlikely to corrupt data by itself. Under concurrent writers, deferred transactions can still surface `SQLITE_BUSY`, uniqueness errors, or 500s instead of clean conflict responses. The object-cleanup race above is also more plausible when a failed deferred writer cleans up after bytes were promoted.

Recommendation: use `BEGIN IMMEDIATE` for short canonical write transactions that do read-before-write checks, or add targeted retry/conflict mapping around known races. Add stress tests for lock acquisition, same-name create/upload completion, folder create/move/rename, TTL sweep versus user mutation, and simultaneous same-digest upload completion.

### Admin Invariants - Admin safety guards are not atomic with the mutations they protect

The admin layer tries to prevent removing the last active administrator, but the checks are separate from the writes they protect. `update_user` fetches the target, calls `ensure_not_last_active_admin`, and then updates `vault_users` with a separate statement (`vault/server/src/admin.rs:208-235`, `vault/server/src/admin.rs:489-507`). Group rename/delete/member-removal paths similarly compute whether an active admin would remain, then perform the group mutation afterward (`vault/server/src/admin.rs:257-318`, `vault/server/src/admin.rs:344-378`, `vault/server/src/admin.rs:510-547`).

The folder-permission guard on group deletion has the same shape. `delete_group` rejects a group if a `folder_permissions` row already references it, but the `SELECT` and `DELETE` are separate statements (`vault/server/src/admin.rs:293-318`). `update_folder_permissions` validates group ids before opening its own transaction, then replaces permission rows inside that transaction (`vault/server/src/folders.rs:445-510`). Because `folder_permissions.group_id` is `ON DELETE CASCADE`, a group delete that passed its stale precheck can later delete a group and cascade-remove permissions that a concurrent folder-permissions update just inserted (`vault/server/src/db/schema.rs:74-84`). The serial route test covers only the already-referenced case, not this race (`vault/server/tests/routes_admin.rs:768-787`).

Impact: two concurrent admin operations can each observe that another active admin still exists and then both commit, leaving no active admin according to the stored flags/groups. A concurrent group delete can also bypass the user-visible "Group is used by folder permissions" guard, deleting the group and revoking freshly committed folder permissions/memberships. These do not corrupt document bytes, but in an OIDC deployment they can create an administrative lockout, surprise authorization loss, or force manual database repair, especially if bootstrap-admin email/group settings do not still grant access.

Recommendation: wrap each admin safety check and its mutation in a single `BEGIN IMMEDIATE` transaction, or make the mutation conditional on a fresh active-admin/permission-use count evaluated inside the same transaction. If permission use is meant to block group deletion, prefer a restrictive FK or guarded delete over `ON DELETE CASCADE`. Add concurrent tests for two admins disabling/demoting each other, group membership removal racing with group rename/delete, and group deletion racing with folder permission assignment.

### Storage / S3-R2 - Remote upload completion stages the entire object under container `/tmp`

For S3/R2, `put_part_files` does not stream parts directly to remote multipart upload. It first calls `stage_part_files`, which writes all part files into a single temporary file under `std::env::temp_dir()` and only then calls `put_file` (`vault/server/src/storage.rs:899-915`, `vault/server/src/storage.rs:1157-1187`). The Dockerfile and compose file only declare/mount `/data`; they do not mount, size, or clean `/tmp` (`Dockerfile:21-51`, `docker-compose.yml:74-78`). The configured transfer path is `/data/transfers`, but this S3/R2 staging temp file is outside that path (`docker-compose.yml:17`).

Impact: with the default 5 GiB upload limit, remote-storage deployments need enough writable container temp space for the full completed upload in addition to `/data/transfers`. If `/tmp` is small, remote uploads fail late at completion. If the process crashes during staging, `vault-s3-upload-*.tmp` leftovers are outside transfer recovery/sweep cleanup and can fill the container filesystem.

Recommendation: stage remote completion under the configured transfer/data directory, document required free space, and clean old `vault-s3-upload-*` files on startup. Prefer true remote multipart upload or a streaming upload body that does not require an extra full-size temp copy.

### Storage Migration - Blob location reads are not filtered to the configured backend

The schema allows multiple `blob_locations` for a single blob, and upload/export completion inserts a new location for the backend that stored the object (`vault/server/src/db/schema.rs:107-115`, `vault/server/src/uploads.rs:1856-1912`, `vault/server/src/exports.rs:1580-1601`). Reads do not select a location compatible with the active storage backend. Document/version downloads join all locations, prefer `local`, and take the first row (`vault/server/src/documents.rs:478-558`). Export artifact downloads also take the first artifact/location row without backend filtering (`vault/server/src/exports.rs:736-794`). The HTTP response and export ZIP writer then ask the single configured `state.storage` backend to read that selected location, and the storage trait rejects backend/bucket mismatches (`vault/server/src/http.rs:1206-1210`, `vault/server/src/http.rs:2274-2314`, `vault/server/src/exports.rs:1218-1230`, `vault/server/src/exports.rs:1322-1331`, `vault/server/src/storage.rs:124-160`).

Impact: storage-backend changes or partial migrations are brittle. If a blob has both local and S3/R2 locations, a deployment configured for S3/R2 can still select the local row and fail with `BackendMismatch` instead of using the readable remote row. If an operator accidentally starts production once with `VAULT_STORAGE_BACKEND=local` and later corrects it to S3/R2, existing metadata remains selectable but unreadable. This can break downloads and exports without any database corruption.

Recommendation: either document storage backend changes as unsupported and fail startup when existing `blob_locations.backend` values do not match the configured backend, or make download/export location selection backend-aware with fallback to another matching, readable location. A migration tool should copy/verify objects, insert current-backend locations, and only then retire old locations.

### Operations - Export queue admission has no quota or backpressure

`VAULT_EXPORT_WORKERS` limits how many export jobs actively build ZIPs, but it does not limit how many jobs can be accepted. `/api/exports` and multi-item `/api/download` both call export job creation (`vault/server/src/http.rs:1199-1243`). `create_export_job_inner` validates the current selection, inserts a `queued` row, and unconditionally calls `start_export_job` for that job (`vault/server/src/exports.rs:511-580`). The worker semaphore is acquired inside the spawned task, so every accepted queued job still creates an in-memory Tokio task waiting for a worker slot (`vault/server/src/exports.rs:622-641`). On startup, transfer recovery can enqueue up to 1000 pending export rows again before serving traffic (`vault/server/src/transfers.rs:13-14`, `vault/server/src/transfers.rs:253-274`). Export artifacts are full ZIP files written under the transfer path and promoted to blob storage until export TTL/sweep cleanup (`vault/server/src/exports.rs:1134-1215`, `vault/server/src/exports.rs:1382-1452`, `vault/server/src/transfers.rs:306-324`, `vault/server/src/transfers.rs:438-473`).

There is also no per-job item cap. Folder export resolution collects every readable descendant document id/download into vectors before building the ZIP (`vault/server/src/exports.rs:860-940`, `vault/server/src/exports.rs:954-963`), and ZIP creation keeps a `Vec` of central-directory entry metadata plus a `HashSet` of written names until finalization (`vault/server/src/exports.rs:1134-1215`). After the artifact is persisted, export history insertion loops once per exported document (`vault/server/src/exports.rs:1539-1563`).

Impact: an authenticated user with read access can queue an unbounded number of expensive exports over the same readable data, and a single export over a very large folder tree can become a large memory, CPU, SQLite-history, transfer-disk, and object-storage job even when `VAULT_EXPORT_WORKERS=1`. Worker concurrency stays bounded, but SQLite rows, spawned waiting tasks, transfer temp files, promoted ZIP artifacts, and object storage can grow until TTL cleanup catches up. After a restart, a large queued backlog is re-spawned before the server starts accepting normal traffic. This is not direct metadata corruption, but it can starve legitimate transfers, fill disk/object storage, and make the service appear hung under normal production deployment.

Recommendation: add global and per-user limits for queued/running export jobs, per-job item counts, per-job/export bytes, and retained export bytes. Reject or 429 new jobs before inserting/spawning when limits are reached, and expose admin cancellation/pruning for backlogs. Consider deduplicating identical export requests for the same user while a matching job is queued/running, stream or spill central-directory planning as needed, and make startup requeue obey the same admission/backpressure rules.

### Transfer State - Queued exports beyond the startup replay limit can stay queued forever

Export jobs are persisted as `queued`, but normal job creation immediately spawns an in-process task for only the new row (`vault/server/src/exports.rs:511-580`, `vault/server/src/exports.rs:622-655`). The generic pending-job starter selects `status = 'queued'` with a caller-supplied limit and starts tasks for only that returned set (`vault/server/src/exports.rs:705-733`). In production startup, that path is reached through interrupted-transfer recovery before the listener is bound, using `DEFAULT_STARTUP_EXPORT_LIMIT = 1000` (`vault/server/src/main.rs:31-51`, `vault/server/src/transfers.rs:13-14`, `vault/server/src/transfers.rs:253-284`). I found no periodic queue drainer or post-startup scheduler for queued rows; the only other production call path is creating a brand-new export (`vault/server/src/http.rs:1230-1243`).

Impact: if a crash happens after export rows are inserted but before their tasks run, or if a large queued backlog exists after repeated restarts or operator/database repair, startup will spawn only the first 1000 queued jobs. Any remaining queued rows can sit in `queued` indefinitely, visible to users as exports that never progress. This is not object corruption, but it is exactly the kind of stuck transfer state that makes production recovery unreliable.

Recommendation: make export queue execution durable instead of task-per-row. Run a bounded background scheduler that repeatedly claims queued jobs while capacity is available, and record worker heartbeats/attempts so stuck queued/running jobs are recoverable without relying on another process restart. Apply queue admission limits before insert so the scheduler is not used as unbounded backlog storage.

### Transfer State - Export completion can be marked failed after the artifact is already committed

`complete_export_job` persists the ZIP artifact, and `persist_export_artifact` inserts the artifact row and updates the job to `complete` before control returns to the caller (`vault/server/src/exports.rs:836-841`, `vault/server/src/exports.rs:1382-1452`). Only after that does `complete_export_job` call `record_export_events`, which inserts one document event per exported file (`vault/server/src/exports.rs:839-840`, `vault/server/src/exports.rs:1541-1563`). If those event inserts fail after the artifact commit, the spawned worker treats the whole export as failed and calls `mark_export_failed`; that update matches every non-cancelled status, including the just-completed row (`vault/server/src/exports.rs:641-653`, `vault/server/src/exports.rs:1604-1624`). Export downloads then reject the job because `export_artifact_download` requires status `complete` even though an artifact row and object may exist (`vault/server/src/exports.rs:736-794`).

Impact: a transient SQLite error or event-table failure at the end of a large export can leave a valid ZIP artifact stored but the user-visible job marked `failed`, so the generated export is not downloadable until an operator repairs or removes it. The artifact can also occupy object storage until TTL cleanup handles the terminal export. This does not alter canonical document bytes, but it creates a misleading and unrecoverable transfer state after the expensive part of the export already succeeded.

Recommendation: make final artifact commit, final job status, and optional audit-event insertion atomic, or record audit events before changing the job to `complete`. If event insertion fails after the artifact is committed, preserve `complete` and surface an audit-warning separately rather than demoting the transfer. Narrow `mark_export_failed` to `queued/running/finalizing` so completed jobs cannot be flipped by post-completion bookkeeping.

### Operations - Upload session admission has no scratch-space quota or backpressure

Upload creation enforces a per-session size limit, but it does not enforce any global or per-user count/byte budget for in-progress sessions. `create_upload_session` validates only the requested size against `VAULT_MAX_UPLOAD_BYTES`, creates a transfer directory, and inserts an `active` `upload_sessions` row (`vault/server/src/uploads.rs:390-477`). Each accepted part is written under that session directory before promotion to canonical storage (`vault/server/src/uploads.rs:620-669`). The defaults allow 5 GiB per session and 24-hour transfer TTLs (`vault/server/src/config.rs:40-53`).

Cleanup is bounded and can lag behind admission. Expired transfer sweeping selects at most 250 expired uploads per run, marks active/completing sessions `expired`, removes their scratch directories after the DB commit, and deletes terminal rows on a later sweep (`vault/server/src/transfers.rs:83-114`, `vault/server/src/transfers.rs:286-304`, `vault/server/src/transfers.rs:408-435`). The sweeper runs at startup and then on the configured interval, defaulting to 60 seconds with a 10-second minimum (`vault/server/src/main.rs:31-45`, `vault/server/src/main.rs:55-75`, `vault/server/src/config.rs:69-99`).

Upload preverification adds a smaller in-memory version of the same backlog. Every accepted part schedules the process-local `UploadHashCoordinator`, which creates a per-session hash state (`vault/server/src/http.rs:1352-1369`, `vault/server/src/uploads.rs:303-388`). That state is only removed by `forget` after upload completion (`vault/server/src/uploads.rs:815-818`, `vault/server/src/uploads.rs:321-323`). Abort, expiry, failure, and transfer sweeps clear files or rows but do not prune the coordinator map (`vault/server/src/uploads.rs:831-859`, `vault/server/src/uploads.rs:2076-2121`, `vault/server/src/transfers.rs:408-435`).

Impact: an authenticated user with write access can create many upload sessions and fill `/data/transfers` with partial uploads until TTL cleanup catches up. The same abandoned sessions can also leave process-local preverification state until restart. This does not by itself corrupt committed document metadata, but a full transfer volume and growing abandoned-session state can break legitimate uploads/check-ins and remote-storage completion, and a large expired backlog can make sweeps repeatedly spend time cleaning old work while new work continues to arrive.

Recommendation: add global and per-user limits for active upload sessions, active upload bytes, transfer scratch disk usage, and in-memory upload preverification state. Reject or 429 new sessions before creating directories/rows when limits are reached, expose current transfer scratch usage to admins, prune hash coordinator state on abort/expiry/failure/sweep, and make sweeps drain backlogs predictably without starving normal traffic.

### Operations - Transfer scratch cleanup is DB-row-driven and can leave untracked files behind

Upload session creation creates the scratch directory before inserting the `upload_sessions` row (`vault/server/src/uploads.rs:419-475`). If the later SQLite insert fails after the directory is created, there is no row for transfer sweep/recovery to follow. Other cleanup paths have the inverse problem: completion, abort, expiry, and failure update the row first and then call `clear_upload_session_files`; that helper ignores `remove_dir_all` failures (`vault/server/src/uploads.rs:819-825`, `vault/server/src/uploads.rs:831-859`, `vault/server/src/uploads.rs:2076-2121`, `vault/server/src/uploads.rs:2389-2392`).

There is also a database-cascade path that can remove the row without running transfer cleanup. Check-in upload sessions store `document_id`, and the schema declares that foreign key with `ON DELETE CASCADE` (`vault/server/src/db/schema.rs:193-209`). Permanent document deletion is a plain `DELETE FROM documents WHERE id = ?` after stale archive/lock checks (`vault/server/src/documents.rs:324-349`). If a check-in upload session still has part files under `transfers/uploads/{session_id}` when the document row is deleted, SQLite can cascade-delete the `upload_sessions` row and leave the scratch directory with no row for `sweep_expired_transfers` or startup recovery to find.

The periodic transfer sweeper has the same DB-first shape. It commits expired/deleted upload and export row changes before removing upload directories and export temp files (`vault/server/src/transfers.rs:103-124`). Terminal upload rows are deleted inside the transaction, then their directories are removed afterward (`vault/server/src/transfers.rs:408-435`); if directory removal fails or was previously skipped, future sweeps have no database row to retry from. I did not find a maintenance path that scans `transfers/uploads/*` or `transfers/exports/*.zip.tmp` for directories/temp files that have no corresponding row.

Impact: failed DB inserts, permission/volume errors during cleanup, document-delete cascades, container interruption between DB cleanup and filesystem cleanup, or ignored upload-directory deletion failures can leave scratch files that are no longer visible to upload/session APIs or sweeps. Over time those untracked files can fill `/data/transfers`, causing new uploads, check-ins, and remote-storage completions to fail even though the database appears clean.

Recommendation: make scratch cleanup durable and observable. Create the DB row before the directory or remove the directory on insert failure, keep terminal rows until filesystem cleanup succeeds, record cleanup errors for retry, and add an orphan-scratch scanner for `transfers/uploads` and `transfers/exports`. Surface untracked scratch usage in readiness/admin maintenance output.

### Operations - Opt-in benchmark sink is an unauthenticated unbounded request-body endpoint

The production router always registers `PUT /api/bench/sink` (`vault/server/src/http.rs:130-142`). The handler returns 404 unless `VAULT_BENCH_ROUTES` is truthy, but when enabled it does not call `current_user` or any admin/dev guard. It streams the entire request body, optionally hashes it, and optionally writes it to a unique file under `VAULT_BENCH_SINK_DIR` or the container temp directory (`vault/server/src/http.rs:287-354`). I did not find this flag in `.env.example`, `docker-compose.yml`, or README production guidance, so the default path is not exposed; the risk is accidental enablement on an internet-facing deployment.

Impact: if benchmark routes are enabled outside an isolated benchmark environment, any unauthenticated client that can reach the container can keep connections busy with arbitrarily large bodies. With `VAULT_BENCH_SINK_WRITE=1`, clients can also fill the container filesystem outside `/data`, bypassing the normal upload size/session/auth controls and transfer cleanup.

Recommendation: require admin auth and dev mode for benchmark routes, or bind them only in test/bench builds. At minimum, document that `VAULT_BENCH_ROUTES` must never be enabled on public deployments and enforce a request-size cap plus temp-file cleanup when the write sink is used.

### Operations - HTTP ingress has no app-level slow-request or concurrency guard

The binary binds a listener and serves the router directly with `axum::serve(listener, app)` (`vault/server/src/main.rs:47-51`). The production router wraps routes with security headers and optional gzip, but I did not find a timeout, load-shed, concurrency-limit, or request-body-rate layer around it (`vault/server/src/http.rs:130-193`). Upload part requests are raw `Request` bodies; the handler acquires a per-session/part process lock and then streams the body into `uploads::ingest_upload_part...` until EOF (`vault/server/src/http.rs:1334-1369`). The ingest path enforces expected offset/size and rejects oversized chunks, but it still waits on the inbound stream while writing the temporary part file (`vault/server/src/uploads.rs:608-669`, `vault/server/src/uploads.rs:1513-1554`).

Impact: with a direct container exposure, or with nginx/TLS configured with permissive buffering/timeouts, slow or stalled clients can tie up request tasks, upload part locks, and temporary part files without any application-side deadline. The normal upload size limit protects completed part size, not connection occupancy or aggregate concurrent slow bodies. This is mainly an availability and deployment-hardening issue, but it can make uploads look hung and can amplify the transfer scratch/backpressure problems above.

Recommendation: add explicit server-side request timeouts/concurrency limits for non-SSE routes and a slow-body/idle timeout for upload and benchmark streaming routes, with SSE/download exceptions where appropriate. Document matching nginx `proxy_read_timeout`, `proxy_send_timeout`, request buffering, and connection limits so production does not rely on unspecified defaults.

### Transfer Lifecycle - Upload part tokens bypass current session status checks

Authenticated part uploads fetch the current user, then fetch the upload session from SQLite, verify owner, and ensure the session is active (`vault/server/src/http.rs:1362-1364`, `vault/server/src/uploads.rs:590-605`). Token-authenticated part uploads skip both live checks: the route verifies the signed token and calls `ingest_upload_part_with_token`, which builds an active session row from token claims (`vault/server/src/http.rs:1358-1362`, `vault/server/src/uploads.rs:578-587`). Token validation checks signature, session id, expiry, owner, mode, size, chunk, and part count, but not the live session row/status or current user `is_active` state (`vault/server/src/uploads.rs:2160-2227`).

Impact: a still-valid upload token can write or recreate scratch part files after the session has been aborted, completed, failed, expired, or after the owner has been disabled. Completion still requires the canonical DB session to be active, so this should not directly corrupt document metadata, but it can waste disk and recreate transfer files after cleanup until token/session expiry.

Recommendation: keep the stateless fast path if needed, but include a session status/version claim that changes on abort/complete/fail, or add a cheap terminal-status check for token-authenticated PUTs.

### Deployment Docs - nginx/TLS upload requirements are not specified

The README says production can run behind a trusted reverse proxy and mentions forwarded proto handling (`README.md:21-23`), but I did not find production nginx guidance for large/resumable uploads: `client_max_body_size`, request buffering, proxy timeouts, and explicit forwarded-header overwrites. The app also relies on an SSE endpoint at `/api/events/stream`; the client opens `EventSource("/api/events/stream")`, while the server keeps an open SSE stream and excludes SSE from gzip compression (`vault/client/src/lib/useVaultResources.js:430-479`, `vault/server/src/http.rs:1047-1053`, `vault/server/src/http.rs:2178-2246`, `vault/server/src/http.rs:200-204`). The README does not specify nginx buffering, read-timeout, or connection handling for that stream. The performance notes also call out missing nginx/container benchmark mode for the deployed topology.

Impact: a nominal production deployment can fail or stall large uploads because of nginx defaults, accidentally forward spoofable headers, or silently buffer/drop SSE state invalidations. Broken SSE does not corrupt the database, but it can leave users operating from stale folder/document/transfer state until a manual refresh and can compound the missed-initial-event issue described above. This is an availability/deployment correctness issue more than an application data corruption issue.

Recommendation: add a minimal nginx/TLS example and a benchmark/readiness checklist for large uploads and SSE. Include `client_max_body_size` aligned with `VAULT_MAX_UPLOAD_BYTES`, deliberate upload buffering behavior, long enough upload/download/SSE proxy timeouts, `proxy_buffering off` or equivalent for `/api/events/stream`, and explicit `proxy_set_header` rules that clear and overwrite forwarded and header-auth identity headers.

## Low

### OIDC Hardening - Public-client mode has no PKCE and multi-audience tokens do not require `azp`

The runtime allows `VAULT_OIDC_CLIENT_AUTH=none` (`vault/server/src/auth.rs:956-968`). The authorization request includes state and nonce, but no `code_challenge`/`code_challenge_method` (`vault/server/src/http.rs:451-473`). ID token validation checks signature, issuer, audience, and nonce (`vault/server/src/oidc.rs:232-267`), and `audience_matches` accepts any audience array containing the client ID without checking `azp` (`vault/server/src/oidc.rs:392-399`).

Impact: confidential server-side clients using `client_secret_basic` or `client_secret_post` are the expected production mode. If operators use public-client mode or an IdP emits multi-audience ID tokens, this is weaker than modern OIDC hardening guidance.

Recommendation: document `client_auth=none` as development/special-case only or add PKCE for that mode. If `aud` is an array with more than one value, require `azp == client_id` when present/required by provider policy.

### Runtime Config - Numeric session/OIDC/transfer settings accept invalid or extreme values

`VAULT_SESSION_MAX_AGE_SECONDS` is parsed with `env_i64` and stored directly without a lower or upper bound (`vault/server/src/auth.rs:205-208`, `vault/server/src/auth.rs:832-845`). Runtime validation checks auth mode, cookie names, cookie secure mode, session secret, public URL, and OIDC client settings, but not session max age (`vault/server/src/auth.rs:250-289`, `vault/server/src/auth.rs:947-987`). The OIDC callback then uses this value both in the signed session `exp` and in the `Set-Cookie Max-Age` attribute (`vault/server/src/http.rs:512-528`, `vault/server/src/http.rs:2751-2761`).

`VAULT_OIDC_NONCE_BYTES` has a lower bound but no upper bound (`vault/server/src/auth.rs:232`). Login uses that value twice, once for OIDC state and once for nonce, and `oidc_token_urlsafe` allocates a byte vector of the requested size before base64 encoding it (`vault/server/src/http.rs:451-461`, `vault/server/src/auth.rs:465-470`).

`VAULT_OIDC_DISCOVERY_TTL_SECONDS` and `VAULT_OIDC_HTTP_TIMEOUT_SECONDS` are also unbounded. The timeout parser rejects non-positive and non-finite values but accepts any finite positive value (`vault/server/src/auth.rs:233-234`, `vault/server/src/auth.rs:847-852`). Discovery cache expiry then uses the configured TTL through saturating timestamp addition, and every OIDC provider request builds a reqwest client with `Duration::from_secs_f64` from the configured timeout (`vault/server/src/oidc.rs:158-188`, `vault/server/src/oidc.rs:340-345`).

The general server config normalizer only enforces lower bounds for upload size/chunk settings, upload/export TTLs, and export worker count (`vault/server/src/config.rs:86-99`). Upload session creation uses those lower-bounded size/chunk values in unchecked chunk-selection and part-count arithmetic before inserting the session row (`vault/server/src/uploads.rs:405-415`, `vault/server/src/uploads.rs:2323-2375`). Upload session creation and export expiry both add configured TTL seconds to `OffsetDateTime::now_utc()` without using checked timestamp arithmetic (`vault/server/src/uploads.rs:421-425`, `vault/server/src/exports.rs:2169-2171`), while export workers are converted to `usize` and used directly as the in-process semaphore capacity (`vault/server/src/exports.rs:143-152`, `vault/server/src/exports.rs:626-641`).

Impact: a zero or negative session max age can create an OIDC login loop by issuing immediately expired/deleted sessions, while an accidentally huge value can keep stale local sessions and group memberships alive far longer than intended. An accidentally huge nonce byte count can make every `/login` request allocate excessive memory before redirecting to the IdP. Extreme OIDC discovery TTLs can preserve stale provider metadata long after an IdP endpoint change, and extreme finite OIDC HTTP timeouts can make login/callback requests wait far longer than intended or panic client construction if the value exceeds `Duration`'s range. Extreme upload size/chunk values can create impractical part counts or hit unchecked integer arithmetic in upload session creation. Extreme upload/export TTL values can also panic request handling when expiry timestamp arithmetic overflows, and an extreme export worker value can remove the intended ZIP-generation concurrency cap. This is configuration hardening rather than a normal request-path corruption bug, but it matters because OIDC authorization, session lifetime, transfer expiry, and export concurrency are production control-plane settings.

Recommendation: validate `VAULT_SESSION_MAX_AGE_SECONDS`, `VAULT_OIDC_NONCE_BYTES`, `VAULT_OIDC_DISCOVERY_TTL_SECONDS`, `VAULT_OIDC_HTTP_TIMEOUT_SECONDS`, upload size/chunk settings, transfer/export TTLs, and `VAULT_EXPORT_WORKERS` at startup with explicit minimum and maximum bounds, and document the security/operability tradeoffs. Use checked timestamp/duration arithmetic and fail startup or reject affected requests with a clear configuration error. For production OIDC, keep the session max age aligned with the desired maximum IdP revocation latency or add live claim refresh.

### Schema Hardening - Several important domains are application-only invariants

The schema has useful foreign keys and uniqueness constraints, but it does not use `CHECK` constraints for status enums, positive sizes/counts, boolean-like fields, TTL actions, or folder acyclicity (`vault/server/src/db/schema.rs:7-23`, `vault/server/src/db/schema.rs:119-174`, `vault/server/src/db/schema.rs:194-282`). Current-version pointers are also application-level metadata: `documents.current_version_id`, `latest_version_number`, and `version_count` are plain columns rather than a constrained pointer to an existing version row (`vault/server/src/db/schema.rs:129-131`). The code validates many of these values on normal request paths, and helper functions tolerate corrupt folder parent cycles without infinite recursion (`vault/server/src/folders.rs:1045-1057`, `vault/server/src/folders.rs:1060-1091`, `vault/server/src/folders.rs:1327-1347`; `vault/server/tests/folders.rs:131-191`). Visible documents with inconsistent current-version metadata cause view builders to fail instead of silently rendering bad rows (`vault/server/src/views.rs:638-648`, `vault/server/src/views.rs:1210-1217`; `vault/server/tests/routes_folders.rs:673-700`).

Impact: this is not a proven normal-request corruption path, but it means future bugs, manual DB edits, or partial migrations can persist impossible states that the database would otherwise reject. For production data safety, status machines and core numeric bounds should be enforced as close to SQLite as practical.

Recommendation: add `CHECK` constraints for transfer statuses, non-negative sizes/progress counters, positive part counts/chunk sizes where applicable, boolean flags, and TTL actions. Enforce current-version pointer consistency with a transaction-local recheck or schema pattern that cannot point at a missing version. Folder acyclicity needs an application-level recheck inside the write transaction or a trigger/recursive CTE guard because SQLite cannot express it as a simple FK.

### Operations - Backup/restore procedure is not documented for SQLite WAL plus object storage

The app stores local runtime state under `/data` and uses SQLite WAL (`README.md:5-13`, `vault/server/src/db/mod.rs:22-27`), but I did not find a production backup/restore procedure covering `vault.db`, WAL/SHM sidecars, local `/data/objects`, `/data/transfers`, or S3/R2 consistency.

Impact: ad hoc file copies can miss WAL contents or capture database metadata out of sync with object files. That is an operational data-loss risk during backup, migration, and restore even if request handling is correct.

Recommendation: document a supported backup flow, ideally SQLite online backup or a controlled maintenance window plus object-store snapshot rules. Include restore validation with `PRAGMA integrity_check`, schema validation, and storage reconciliation.

### Share Lifecycle - Share links cannot be expired, disabled, or listed through the product

Share links have lifecycle fields in the schema (`expires_at` and `disabled_at`), and resolution honors them (`vault/server/src/db/schema.rs:293-308`, `vault/server/src/shares.rs:137-142`, `vault/server/src/shares.rs:309-323`). However, link creation never sets an expiry or disabled timestamp (`vault/server/src/shares.rs:74-124`), and the exposed routes only create and resolve links (`vault/server/src/http.rs:168-169`, `vault/server/src/http.rs:1068-1088`). The client action mirrors that: it posts to `/api/share-links`, copies the URL, and exposes no list/revoke/expiry flow (`vault/client/src/lib/shareLinks.js:200-231`). Tests exercise disabled/expired links by direct SQL updates rather than through an API (`vault/server/tests/routes_shares.rs:313-327`).

Impact: this is not an authorization bypass because share resolution re-checks the current user's access to the target, and target deletion cascades link rows. The production lifecycle problem is that old internal pointers accumulate and cannot be deliberately revoked or aged out by users/admins. A copied link that stopped resolving after access was removed can become usable again for the same recipient if access is later re-granted to the target folder/document.

Recommendation: add list/revoke APIs and UI for share links, support explicit expiry/default TTL, and add a small sweeper or admin maintenance path for expired/disabled rows. Keep the current access re-check, but make link lifecycle auditable and operator-controlled.
